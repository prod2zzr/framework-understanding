# 哪吒模式：一键触发的多智能体并行工作流

---

## 从比喻到架构

用户的原话描绘了一幅完整的工作流画面：

> 多模态模型是**千里眼顺风耳**——信息获取和预处理。
> Claude Code 再化身**三头六臂的哪吒**——同时打开若干终端，指向同一目录，在不同子任务中并行工作。
> 对比结果，给出 summary，为决策者提供科学依据。
> **决策者只点了一下"发送"。**

这不是比喻。这是一个可以用现有技术实现的架构：

```
决策者
  │
  │  点击"发送"
  ↓
┌─────────────────────────────────────────────────────────┐
│  Claude Code（编排者）                                   │
│                                                         │
│  Phase 1: 千里眼顺风耳                                   │
│  ┌─────────────────────────────────────────────┐        │
│  │ 多模态预处理                                  │        │
│  │ · 读取 PDF/图片/表格/代码/音频                │        │
│  │ · 扫描项目目录结构                            │        │
│  │ · 提取关键信息，生成结构化上下文               │        │
│  └─────────────────────┬───────────────────────┘        │
│                        │                                │
│  Phase 2: 三头六臂                                      │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐   │
│  │Terminal 1│  │Terminal 2│  │Terminal 3│  │Terminal 4│   │
│  │子目录 A  │  │子目录 B  │  │子目录 C  │  │子目录 D  │   │
│  │方案探索  │  │数据分析  │  │风险评估  │  │竞品对标  │   │
│  └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘   │
│       │            │            │            │          │
│  Phase 3: 收敛                                          │
│  ┌─────────────────────────────────────────────┐        │
│  │ 对比所有结果 → Summary → 决策建议             │        │
│  └─────────────────────────────────────────────┘        │
│                                                         │
└────────────────────────────┬────────────────────────────┘
                             │
                             ↓
                        决策者审阅
                     （也许再点一下"同意"）
```

## 1. "千里眼顺风耳"：多模态预处理已经成熟

### 过去 vs. 现在

```
过去（2023）：                          现在（2026）：
──────────                             ──────────
· 手动阅读 50 页 PDF                    · Claude 直接读取 PDF，提取结构化数据
· 手动截图 → 描述给 AI                  · 多模态模型直接"看"图片/图表
· 手动整理 Excel → 复制粘贴             · 直接解析 .xlsx/.csv，理解表结构
· 手动搜索 10 个网站                    · WebSearch + WebFetch 并行搜索
· 花 2 小时收集信息                     · 花 2 分钟，或者 0 分钟（自动化）
```

Claude Code 当前的多模态能力：
- **读取 PDF**：直接解析，支持指定页码范围
- **读取图片**：PNG/JPG/截图，视觉理解
- **读取代码**：任意编程语言，理解上下文
- **读取 Jupyter Notebook**：代码 + 输出 + 可视化一体
- **搜索 Web**：实时获取最新信息
- **执行 Bash**：运行脚本处理数据

**预处理的本质**：把多种格式的原始信息，收敛为结构化的上下文文件——这正是"千里眼顺风耳"到"收敛"的桥梁。

## 2. "三头六臂"：Claude Code 的并行执行能力

### 三种并行机制

Claude Code 已经具备三种"分身"能力，对应不同的并行粒度：

#### 机制一：并行工具调用（最轻量）

单个 Claude Code 会话中，同时发起多个独立的工具调用：

```
单次响应中同时执行：
├── [Grep]  搜索子目录 A 中的关键模式
├── [Grep]  搜索子目录 B 中的关键模式
├── [Read]  读取配置文件 X
└── [Read]  读取配置文件 Y

全部并行，结果一起返回。
```

适合：信息收集阶段的快速扫描。

#### 机制二：Sub-Agent（中等粒度）

通过 Agent 工具启动独立的子智能体，每个有自己的上下文窗口：

```
主 Claude Code 会话
├── Agent(subagent_type="Explore")   → 探索子目录 A 的架构
├── Agent(subagent_type="Explore")   → 探索子目录 B 的架构
├── Agent(subagent_type="Plan")      → 规划方案 C
└── Agent(isolation="worktree")      → 在隔离副本中实验方案 D

每个子智能体独立工作，互不干扰。
主智能体等待所有结果返回后，综合分析。
```

关键参数：
- `run_in_background: true`：后台运行，不阻塞其他工作
- `isolation: "worktree"`：Git worktree 隔离，拥有仓库的独立副本
- `model: "sonnet"/"opus"/"haiku"`：可以为不同子任务选择不同模型

适合：需要深入分析的多维度研究。

#### 机制三：Agent Teams（最重量级）

多个完整的 Claude Code 实例协同工作：

```bash
# 启用实验性 Agent Teams
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1

# 一个编排者 + 多个执行者
# 通过 tmux 可以同时观察所有智能体的工作
```

特性：
- 每个智能体是完整的 Claude Code 实例
- 独立的上下文窗口、工具权限、工作目录
- 智能体间可直接通信（不必经过编排者）
- 通过 tmux 界面人类可实时观察和介入
- 编排者用 Opus，执行者可用 Sonnet（成本优化）

适合：大型复杂项目的全面并行处理。

### 三种机制的对比

| 维度 | 并行工具调用 | Sub-Agent | Agent Teams |
|------|:----------:|:---------:|:-----------:|
| 并行度 | 3-5 个工具 | 5-10 个子智能体 | 16+ 个完整实例 |
| 上下文隔离 | 共享 | 独立 | 完全独立 |
| 通信方式 | 同一会话内 | 返回结果给主智能体 | 智能体间直接通信 |
| 人类介入 | 每步审查 | 结果审查 | tmux 实时观察 |
| 成本 | 1x | 4-8x | 15x+ |
| 适用场景 | 信息收集 | 多维分析 | 大型项目 |

## 3. 实操：一键触发的完整工作流

### 场景：为决策者准备竞品分析报告

决策者的全部输入：

```
"分析我们的产品和 3 个主要竞品在安全性、性能、用户体验方面的差异，
 给出选型建议。参考资料在 /data/ 目录中。"
```

一下"发送"。以下是 Claude Code 内部发生的事情：

### Phase 1: 千里眼顺风耳（自动，~2 分钟）

```
Claude Code 自动执行：

1. [Glob] 扫描 /data/ 目录结构
   → 发现: product-a.pdf, product-b.xlsx, product-c-screenshots/,
     our-product-docs/, benchmark-results.csv

2. [并行读取]
   ├── [Read] product-a.pdf (pages: "1-10")  → 提取产品 A 核心特性
   ├── [Read] product-b.xlsx                  → 解析产品 B 数据表
   ├── [Read] product-c-screenshots/*.png     → 视觉分析产品 C UI
   ├── [Read] our-product-docs/               → 读取我方产品文档
   └── [Read] benchmark-results.csv           → 解析性能基准数据

3. [WebSearch] 并行搜索最新信息
   ├── "Product A security audit 2026"
   ├── "Product B performance benchmark"
   └── "Product C user reviews 2026"

4. 生成结构化上下文 → 写入临时 context 文件
```

### Phase 2: 三头六臂（并行，~5 分钟）

```
Claude Code 同时启动 4 个子智能体：

Agent-1 (安全性分析):
  "基于以下上下文，对比 4 个产品的安全架构。
   评估维度：加密标准、认证机制、漏洞历史、合规认证。
   输出：结构化对比表 + 风险评级。"

Agent-2 (性能分析):
  "基于 benchmark-results.csv 和搜索结果，对比 4 个产品的性能。
   评估维度：响应时间、吞吐量、资源消耗、扩展性。
   输出：性能对比表 + 图表描述。"

Agent-3 (用户体验分析):
  "基于截图和用户评价，对比 4 个产品的用户体验。
   评估维度：易用性、学习曲线、界面设计、用户满意度。
   输出：UX 评分矩阵 + 关键差异点。"

Agent-4 (综合对标):
  "基于以下上下文，分析 4 个产品的市场定位和定价策略。
   评估维度：价格、目标客户、市场份额、增长趋势。
   输出：竞争力雷达图数据 + SWOT 分析。"

所有子智能体并行运行，互不干扰。
```

### Phase 3: 收敛与 Summary（~2 分钟）

```
所有子智能体返回结果后，主 Claude Code 自动执行：

1. 整合 4 份分析报告
2. 交叉验证数据一致性（安全分析中的数据 vs 性能分析中的数据）
3. 识别冲突点和盲点
4. 生成决策摘要：

   ┌────────────────────────────────────────────┐
   │  竞品分析决策摘要                            │
   │                                            │
   │  推荐：产品 B                               │
   │  理由：安全性最优 + 性能第二 + 价格合理      │
   │  风险：UX 评分最低，需要额外培训投入          │
   │                                            │
   │  备选：维持现有产品                          │
   │  理由：切换成本高，当前满足 80% 需求          │
   │                                            │
   │  详细数据：见下方对比表                      │
   │  完整报告：见 /output/ 目录                  │
   └────────────────────────────────────────────┘
```

### 决策者收到的

从"发送"到收到结果：**~9 分钟**。

过去同样的工作量：一个分析师需要 **2-3 天**。

## 4. 让"一键"成为可能的前提：预配置

"决策者只点一下发送"之所以能实现，是因为所有的收敛工作已经**前置**了：

```
前置投入（一次性，30分钟）          "一键"之后（每次，自动）
─────────────────────             ────────────────────
CLAUDE.md 定义了：                 Claude Code 自动知道：
· 公司名称和产品定位                · 分析的参照系是什么
· 输出语言和格式偏好                · 用中文、Markdown 表格
· 竞品清单和关注维度                · 分析哪些竞品、哪些维度
· 数据源位置                       · 去哪里找原始数据
· 质量标准                         · 什么程度算"够好"

.claudeignore 排除了：              Claude Code 自动忽略：
· 无关的历史存档                    · 不浪费 token 在噪音上
· 敏感的财务细节                    · 不泄露不该看的数据

Context files 提供了：              Claude Code 自动具备：
· 行业术语表                       · 正确使用专业术语
· 分析框架模板                      · 按标准框架输出
· 历史决策记录                      · 理解决策上下文
```

**这就是整个系列的实操汇合点**：
- 第 5 篇的 SOP（用 Code 构建上下文架构）→ 完成前置配置
- 第 7 篇的收敛理论 → 解释了为什么前置配置有效
- 第 8 篇的多智能体编排 → 解释了"三头六臂"的技术基础
- **本篇**：把所有理论变成一个端到端的工作流

## 5. 进阶：让哪吒更聪明的编排策略

### 策略一：先竞争后协作

```
同一个问题，让 3 个子智能体用不同方法分析：

Agent-A: 定量分析（纯数据驱动）
Agent-B: 定性分析（专家判断 + 案例对比）
Agent-C: 反面分析（专门找"不应该选"的理由）

然后主智能体综合三个视角 → 更全面的结论
```

这就是"竞争"的实际价值：不是为了分出胜负，而是**消除单一视角的盲点**。

### 策略二：分层收敛

```
Level 1: 4 个子智能体各自分析 → 4 份报告
Level 2: 2 个子智能体各审查 2 份报告 → 2 份审查意见
Level 3: 1 个主智能体综合所有输出 → 1 份最终报告

每一层都是收敛。从 4 → 2 → 1。
```

### 策略三：人机混合编排

```
Phase 1: 全自动（千里眼 + 三头六臂）
Phase 2: 人类审查中间结果（30 秒快速浏览）
Phase 3: 人类微调方向（"深入分析产品 B 的安全问题"）
Phase 4: 自动执行深入分析
Phase 5: 人类最终裁判

总人类时间：~3 分钟（审查 + 微调 + 裁判）
总机器时间：~15 分钟（并行工作）
```

## 6. 从个人到组织：规模化的"一键"

### 个人决策者

```
一个人 + Claude Code = 一个分析团队的产出
```

### 团队协作

```
团队共享 CLAUDE.md + context files（通过 Git）
每个成员用同样的上下文启动自己的 Claude Code
输出一致性由共享配置保证

                 共享的 CLAUDE.md
                       │
         ┌─────────────┼─────────────┐
         ↓             ↓             ↓
    成员 A 的        成员 B 的       成员 C 的
    Claude Code     Claude Code    Claude Code
    (分析市场)      (分析技术)     (分析财务)
         │             │             │
         └─────────────┼─────────────┘
                       ↓
               汇总到共享目录
                  → 决策者审阅
```

### 自动化流水线

最终形态——不是人点"发送"，而是**事件触发**：

```
触发条件                    自动执行
──────────                 ──────────
· 新数据文件出现在 /data/   → 自动启动分析流水线
· 竞品发布新版本            → 自动启动对标分析
· 周一早上 9:00             → 自动生成周报摘要
· PR 合并到 main            → 自动运行安全审计

人类只在"异常"时被通知。
常规决策完全自动化。
```

## 7. 代价与边界

### Token 经济学

```
工作流阶段              Token 消耗          成本估算
──────────             ──────────          ──────────
Phase 1 (预处理)       ~20K tokens         ~$0.30
Phase 2 (4 并行)       ~80K tokens × 4     ~$4.80
Phase 3 (收敛)         ~30K tokens         ~$0.45
──────────────────────────────────────────────────
单次完整分析            ~350K tokens        ~$5.55

对比：人工分析师 2-3 天 × 日薪 = 数千元
ROI: 当分析频率 > 每月 1 次，自动化即回本
```

### 不适合"一键"的场景

| 场景 | 原因 | 应该怎么做 |
|------|------|-----------|
| 首次接触全新领域 | 人类还不知道"什么重要" | 先用 Code 交互式探索，建立 context |
| 政治敏感的决策 | 需要理解组织动态和人际关系 | AI 提供数据分析，人类做政治判断 |
| 创意/品牌工作 | 审美和品牌调性难以量化 | AI 生成选项，人类做审美选择 |
| 法律/合规判断 | 需要专业资质和法律责任 | AI 做研究和初筛，专业人士做判断 |

**规律**：越是可以结构化、量化、有明确评判标准的任务，越适合"一键"。越依赖隐性知识、人际判断、审美感受的任务，越需要人类深度参与。

## 8. 完整的画面：从第 1 篇到第 9 篇

本系列走过的路：

```
第 1-3 篇: 理解单个工具
           "这些工具的架构是什么？"
                    │
第 4 篇:   发现问题
           "为什么 Cowork 不好控？"
                    │
第 5 篇:   解决问题
           "用 Code 为 Cowork 建上下文架构"
                    │
第 6 篇:   放大视野
           "整个市场在发生什么？"
                    │
第 7 篇:   提炼本质
           "人类的核心角色：收敛"
                    │
第 8 篇:   扩展架构
           "不止一个 AI——协作与竞争"
                    │
第 9 篇:   落地实操（本文）
           "千里眼 + 三头六臂 + 一键发送"
```

最终的统一模型：

```
决策者（人类）
  │
  │  一键发送（+ 前置配置的收敛投入）
  ↓
千里眼顺风耳（多模态预处理）
  │
  │  结构化上下文
  ↓
三头六臂（多智能体并行）
  │
  │  协作 + 竞争
  ↓
收敛为 Summary
  │
  │  科学依据
  ↓
决策者审阅（也许再点一下"同意"）
```

**从 50 页 PDF 到一个决策建议，中间的所有工作——阅读、分析、对比、综合——都被多个 AI 并行完成。人类只做了两件事：事先定义好"什么是重要的"（配置），事后判断"哪个方案是对的"（裁判）。**

这就是哪吒模式。
