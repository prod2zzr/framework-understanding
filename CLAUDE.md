# 全局开发规范 — 字节范（ByteStyle）

> 以字节范核心价值观驱动高效协作。
> 本文件基于 ByteStyle 模板适配 **AI 合同审核系统（contract-reviewer）** 项目。

### 最高原则：做好比做完重要

> **此原则不可被任何项目级规则覆盖。**

- **质量优先于速度**：做对一次，好过返工三次。"赶紧出结果"不是跳过思考的理由
- **先想后做**：收到需求后的第一反应不是动手写代码，而是确认自己理解了问题。动手前花 1 分钟想清楚，省下 10 分钟返工
- **慢就是快**：该走的流程不跳过、该读的代码不跳过、该问的问题不跳过。每一次"省事"的跳过，都在积累后面更大的返工成本
- **合同审核的特殊约束**：本系统输出具有法律参考价值，错误可能导致实际经济损失，对准确性和可追溯性有更高要求

## 一、工作原则

### 始终创业 (Always Day 1)

- **最简方案优先**：用最少的代码解决当前问题，不为假想需求增加复杂度
- **MVP 驱动**：新功能先跑通最小可用版本，验证方向后再迭代完善
- **垂直切片**：按端到端功能路径拆分，每一片都可独立运行和验证。避免先写完所有模型、再写所有接口的水平分层式开发
- **快速反馈**：每次改动后立即验证，保持短反馈循环
- **敢于丢弃**：如果发现方向错了，果断推翻重来，沉没成本不是继续的理由

### 坦诚清晰 (Be Candid and Clear)

- **commit message**：`<type>(<scope>): <中文简要描述>`
  - type: feat / fix / refactor / docs / test / chore / perf / ci
  - scope: llm / rag / review / chunking / plugins / ocr / api / cli / config / docs
- **代码即文档**：命名自解释，注释说明"为什么"而非"做了什么"
- **主动暴露问题**：遇到不确定的方案直接提问，不猜测不硬上
- **有意义的错误信息**：错误提示要能帮助定位问题，不吞异常

### 求真务实 (Seek Truth and Be Pragmatic)

> 回归第一性原理：从问题的基本事实出发推导方案，不从"上次怎么做的"或"别人怎么做的"出发类比。每一个"理所当然"都值得追问一句"为什么"。

- **先读后改**：修改代码前必须先理解现有实现，搞清楚上下游影响
- **先理解再删除**（Chesterton's Fence）：删除或替换代码前，先搞清楚它为什么存在（查 git blame、调用链、测试）。看不懂的代码不等于没用的代码
- **根因驱动**：bug 修复先复现、定位根因，再写修复代码，不做表面补丁
- **数据说话**：性能优化基于实际瓶颈，不做无依据的"优化"
- **保持整洁**：删除无用代码，不留注释掉的废弃代码
- **第二次出现时统一**：同一逻辑第一次出现直接写，第二次出现提取为共享工具（如 `utils/`），不等到第三次扩散

### 多元兼容 (Be Open and Inclusive)

- **尊重现有风格**：修改代码时保持与项目已有模式一致，不擅自按个人偏好重构
- **兼容性意识**：考虑云端（Claude API）和本地（Ollama）双部署模式
- **依赖克制**：引入新依赖前先问"项目里有没有现成的"

### 敢为极致 (Be Courageous and Aim for the Highest)

- **安全第一**：防范 OWASP Top 10，边界输入必须校验，外部数据不可信
- **敏感信息零容忍**：密钥、token、密码禁止硬编码或提交，全部走 `CR_` 环境变量
- **测试护航**：核心业务逻辑必须有测试覆盖，验证脚本必须沉淀为可回归的自动化用例。写超过 100 行代码前必须先跑一次测试
- **测试验结果不验内部**：断言输出和行为，不断言方法调用次数。优先用真实实现，只在真正的系统边界使用 mock
- **验证才算完成**：功能开发完成后必须运行自动化验证，通过后才能进入收尾流程。"应该可以"和"看起来对了"不算验证——必须有可执行的验证命令和通过的输出
- **不将就**：发现代码有隐患时当场提出，不留"以后再说"的隐患

### 共同成长 (Grow Together)

- **知识沉淀**：关键决策和经验**必须记录到 `docs/` 目录并随 git 提交**，memory 仅作为跨会话辅助索引。文档是团队共享的持久化真相源，memory 是个人的临时记忆
- **小步提交**：大型改动拆分为小的、可独立审查的 PR。单次提交 ~100 行为宜，超过 300 行应考虑拆分，超过 1000 行必须拆分
- **复盘驱动成长**：每次大任务完成后复盘，从实战中沉淀经验
- **大改动后质量回合**：超过 200 行的特性 PR 后，紧跟一个专项质量检查 PR（竞态、边界、性能）
- **测试即资产**：开发过程中编写的验证脚本必须沉淀为回归测试用例，不允许用完即弃

## 二、通用技术规范

### 2.1 语言与风格

- **Python 3.11+**，充分使用现代语法（`X | Y` union types、`match/case`、`ExceptionGroup`）
- **Type hints**：所有 public 函数必须完整标注参数和返回类型
- **Pydantic v2**：所有数据模型继承 `BaseModel`，配置继承 `BaseSettings`
- **async/await**：I/O 密集操作（LLM 调用、文件读写、embedding）必须异步
- **Docstrings**：每个模块、类、public 函数一行 `"""English summary."""`（与现有风格一致）
- **Imports**：stdlib → third-party → local，每组内字母序。由 ruff `I` 规则自动排序
- **日志**：`logger.info("msg %s", arg)` 格式，**禁止** f-string in log calls
- 代码注释和文档使用中文，变量名和代码使用英文
- 文件末尾保留空行，UTF-8 编码

### 2.2 项目架构约定

```
src/contract_reviewer/
├── app.py           # FastAPI 入口（API 模式）
├── cli.py           # Typer 入口（CLI 模式）
├── models/          # Pydantic 数据模型 + Settings
│   └── config.py    # pydantic-settings, CR_ env prefix, settings.yaml fallback
├── llm/             # LiteLLM 封装 + 韧性层（重试/熔断/预算）
├── rag/             # 向量检索：embedder + vectorstore + ingestor
├── review/          # 审核引擎 + 审计追踪 + 验证器
├── chunking/        # 合同解析（PDF/DOCX/TXT）+ 分块
├── plugins/         # 插件系统（ReviewPlugin 基类 + 内置插件）
├── ocr/             # OCR 支持（PaddleOCR）
├── streaming/       # SSE 流式推送
└── utils/           # 共享工具（hashing, jsonl）
```

- **Settings** (`CR_` prefix)：所有配置通过 `models/config.py` 的 `Settings` 类管理
- **Prompts**：Jinja2 模板在 `config/prompts/`
- **Rules**：YAML 格式在 `config/rules/`
- **插件**：继承 `plugins.base.ReviewPlugin`，通过 `registry.py` 自动发现

### 2.3 Git 规范

- **Commit 格式**：`<type>(<scope>): <中文简要描述>`
  - type: feat / fix / refactor / docs / test / chore / perf / ci
  - scope: llm / rag / review / chunking / plugins / ocr / api / cli / config / docs
- **分支命名**：`feat/xxx`、`fix/xxx`、`refactor/xxx`
- **重构与功能分离**：重构提交不混入功能变更或 bug 修复，单独 commit / PR 提交
- **原子提交**：一个 commit 只做一件事。格式变更不混入行为变更
- **主分支**通过 PR 合入，不直接推送
- 合并前确保 lint + test 通过

### 2.4 依赖与环境

- **包管理**：`pyproject.toml`（hatchling 构建），单一真相源
- **可选组**：`ocr`（PaddleOCR）、`reranker`（sentence-transformers）、`dev`（pytest）
- `.env` 文件不入版本控制，`.env.example` 作为模板
- 优先复用项目已有的库，引入新依赖前说明必要性

### 2.5 韧性编码规范

> 生产环境中，异常条件出现的频率高于正常完成。错误路径即主路径。

- **LLM 调用**：必须设 120s 超时；仅对 429/5xx 做指数退避重试（1s/2s/4s，最多 3 次），4xx 立即失败
- **并发资源**：Token 预算、Semaphore 等必须用 `async with` 管理，确保异常时也释放
- **缓存 key**：必须包含模型 ID，防止换模型后命中错误缓存
- **编码回退链**：中文文档处理按 UTF-8 → GB18030 → GBK → Latin-1 顺序尝试
- **扫描件检测**：PDF 解析后文本量低于 50 字符时，标记为扫描件走 OCR 路径，不直接送 LLM
- **熔断器**：连续 3 次失败 → OPEN（快速失败）→ 30s 后 HALF_OPEN（试探）→ 成功则 CLOSED
- **回退链模式**：关键路径提供多级回退（如 RAG：预计算向量 → 运行时 embedding → 优雅降级）

### 2.6 项目文档规范

项目使用 `docs/` 目录统一管理文档，流转关系：

```
backlog（想法）→ prd（产品需求）→ designs（UI 交互）→ spec（技术方案）→ plan（拆步骤）→ tests（验证 + 回归）→ decisions（复盘）
```

> 不是每个环节都必须有产出，没有产物的环节跳过即可。

| 目录 | 级别 | 用途 | 命名格式 |
|------|------|------|---------|
| `docs/backlog/` | **必须** | 需求池 & 待办事项 | `YYYY-MM-DD-<主题>.md` |
| `docs/prds/` | 推荐 | 产品需求文档 | `YYYY-MM-DD-<功能名>.md` |
| `docs/designs/` | 推荐 | UI 交互 | `YYYY-MM-DD-<功能名>/` |
| `docs/specs/` | **必须** | 方案设计（背景 + 调研 + 决策 + 技术方案） | `YYYY-MM-DD-<模块名>-<主题>.md` |
| `docs/plans/` | **必须** | 开发计划（任务拆分 + 进度跟踪） | `YYYY-MM-DD-<模块名>-plan.md` |
| `docs/tests/` | **必须** | 测试用例（功能验证 + 回归测试） | `<模块名>-<功能>.md` |
| `docs/decisions/` | **必须** | 复盘与经验教训 | `YYYY-MM-DD-<主题>.md` |
| `docs/guides/` | 推荐 | 开发指南（环境搭建、上手文档） | `<主题>.md` |
| `docs/changelog/` | 建议 | 版本变更记录 | `YYYY-MM-DD-v<版本号>.md` |
| `docs/runbooks/` | 建议 | 运维手册（部署、故障处理） | `<场景>.md` |
| `docs/assets/` | **必须** | 文档资源文件（图表、截图） | `<图表类型>-<描述>.png` |

文档与代码同步更新，中文编写，文件名英文小写+连字符。文档必须随代码一起 git 提交。

**Backlog 索引**：`docs/backlog/INDEX.md` 按优先级分组（P0-P3），`[ ]` = open，`[x]` = done。

## 三、任务工作流

### 完整流程

```
task-start（启动）→ task-execute（执行）→ 自动化验证 → task-finish（收尾）
```

按决策树判断深度：
- **明确小任务**（≤2 文件）：直接动手
- **有模糊点**（≤3 文件）：仅需求对焦，然后直接开发
- **大型任务**（3+ 文件 / 新模块 / 架构决策）：完整走 start → execute → finish
- **跨会话任务**：必须启用 task-execute 框架

### PR 自检清单（代码审计 10 项）

提交前逐项确认：

1. 无静默吞异常（`except: pass`）
2. 无硬编码密钥/token（全部走 `CR_` 环境变量）
3. 无未提取的魔法数字（阈值、权重、窗口大小等）
4. 无死代码或注释掉的废弃代码
5. 异常均携带描述信息
6. 哈希算法统一（SHA-256，使用 `utils/hashing.py`）
7. 无重复逻辑（JSONL 写入用 `utils/jsonl.py`）
8. Import 按规范分组排序
9. 所有 public 函数有完整类型标注
10. 无未使用的 import

### 效率准则

- 能用一句话说清的不用三句话
- 能一次改对的不来回试探
- 能并行处理的不串行等待
- 能复用现有方案的不重新发明
- **先想后动**：想动手时先停 5 秒——我理解需求了吗？我读过相关代码了吗？该走的流程走了吗？

### 范围控制 — 防止 Scope Creep

- **只做被要求的事**：发现"顺手可以优化一下"是效率杀手，记下来但不现在做
- **改动最小化**：每次 PR 只包含一个目的的改动，不混入无关重构
- **识别膨胀信号**：当发现"这个也要改、那个也要动"时，停下来重新评估范围

### 问题升级与止损

- **Stop-the-line**：测试失败时停止一切新功能开发，先修再走。不允许"先跳过这个失败的测试继续下一个功能"
- **三次原则**：同一个思路尝试 3 次仍不通，立即换方向或求助
- **主动升级**：复杂度超预期、信息不足、需要决策时立即告知用户
- **六步排障法**：复现（不能复现 = 不能自信修复）→ 定位层级（UI / API / DB / 外部服务）→ 最小化（剥离无关代码）→ 修根因（五个为什么）→ 加回归测试（没有 fix 时必须失败）→ 在调用位置或服务边界加同样守卫（让 bug 无法重新引入）→ 端到端验证

### 风险分级执行

| 风险等级 | 场景示例 | 策略 |
|---------|---------|------|
| **低** | 修改局部变量、改样式 | 直接执行，改后验证 |
| **中** | 修改公共函数签名、加依赖 | 先评估影响，改后跑测试 |
| **高** | 数据库 migration、删除文件/分支 | 与用户确认后再执行 |
| **极高** | 生产环境操作、force push | 必须用户明确授权 |

## 四、常见反模式

| 反模式 | 问题 | 正确做法 |
|--------|------|---------|
| 改之前不读 | 基于假设修改，引入新 bug | 先读完相关代码再动手 |
| 一步到位 | 大量改动一起提交，难定位问题 | 小步提交，逐步验证 |
| 过度设计 | "万一以后需要"导致复杂度膨胀 | 只解决当前问题 |
| 顺手重构 | 改 A 功能时顺手重构了 B 模块 | 记下来开新任务 |
| 沉默卡死 | 卡住了还在默默尝试 | 3 次不通就换方向或求助 |
| 表面补丁 | 不查根因，哪里报错补哪里 | 先定位根因 |
| 假装完成 | 核心路径没验证就说"改好了" | 必须运行验证 |
| 赶着交付 | 跳过思考急于出结果 | 先想清楚再动手 |
| f-string in log | `logger.info(f"x={x}")` 性能浪费 | `logger.info("x=%s", x)` |
| 同步阻塞异步 | 在 async 函数中调用阻塞 I/O | 使用 `asyncio.to_thread` 或原生异步 |
| 裸 Exception | `except Exception: pass` | 捕获具体异常（LLMError, ParseError 等） |
| 硬编码模型名 | `model="claude-xxx"` | 使用 `Settings.llm_model` |
| 只存 memory | 重要决策只记 memory 不入库 | 关键信息写入 `docs/` 随 git 提交 |
| 验证即弃 | 验证脚本用完就删 | 沉淀到 `docs/tests/` 作为回归用例 |
| 假设部署环境 | 以为"本地"就是 GPU 服务器 | 先调研真实用户的硬件和网络约束 |
| 串行处理可并发 I/O | Embedding 逐条串行处理 | `asyncio.gather` + `Semaphore` 并发 |
| 实例变量存并发状态 | 审计日志写入竞态条件 | 函数级局部变量 + 最终合并 |
| 重构混入功能 | 一个 PR 既改行为又改结构 | 重构和功能变更分开提交 |
| 测试验内部实现 | mock 方法调用次数，重构即挂 | 断言输出和行为，不断言内部调用 |
| 跳过失败的测试继续开发 | 累积隐性债务，最终爆发 | Stop-the-line：先修再走 |
| 水平分层开发 | 先写完所有 model 再写 API | 垂直切片：每片端到端可运行 |
| 凭直觉调试 | "让我试试这个"无假设随机试探 | 先陈述假设，加探针验证，证据矛盾则彻底丢弃假设 |
