## 一、架构决策原则

### 1.1 约束驱动设计，而非功能驱动

项目最成功的架构决策都源于**真实约束**，而非假想需求：

| 约束 | 驱动的设计 | 收益 |
|------|-----------|------|
| 法务人员用 16GB 笔记本 | 分阶段加载模型（OCR→卸载→RAG→LLM） | 峰值内存 = max(单模型) 而非 sum(全部) |
| 法规频繁更新，微调不现实 | RAG + 向量库增量更新 | 业务专家自主管理知识库 |
| 非技术人员需参与规则维护 | 三层配置分离（Jinja2/YAML/环境变量） | 法务直接编辑 YAML 规则 |
| 云端+本地双部署模式 | LiteLLM 统一抽象层 | 一个环境变量切换部署模式 |

**教训**：早期假设"本地部署"意味着 GPU 服务器，实际用户是笔记本。**先搞清楚真实约束再做架构决策。**

### 1.2 配置即接口

```
修改频率低 → Jinja2 模板（开发者改）
修改频率中 → YAML 规则（业务人员改）
修改频率高 → 环境变量（运维改）
```

**核心洞察**：当配置按"谁来改、改多频繁"分层时，非工程师就从系统的"用户"变成了"共建者"。

### 1.3 抽象时机：第二次出现时统一

- 第一次出现：直接写，不抽象
- 第二次出现：提取为共享工具（如 `utils/hashing.py`、`utils/jsonl.py`）
- 第三次才统一：已经太晚，重复代码已扩散

**反例**：哈希逻辑在 3 个文件中各写一遍（且 ingestor.py 用了 MD5），审计时才统一为 SHA-256。

## 二、生产韧性：错误路径即主路径

> 生产环境中，异常条件出现的频率高于正常完成。

### 2.1 防御式编程清单

| 场景 | 防御措施 | 原因 |
|------|---------|------|
| LLM 调用 | 120s 超时 + 指数退避重试（仅 429/5xx） | 防止无限挂起和雪崩 |
| 并发控制 | Token 预算 + Semaphore + `async with` | 确保异常时也释放资源 |
| 缓存 | Key 含模型 ID | 防止换模型后命中错误缓存 |
| 扫描件 PDF | 50 字符最低阈值检测 | 空文本会导致 LLM 幻觉 |
| 中文编码 | UTF-8 → GB18030 → GBK → Latin-1 回退链 | 商业文档编码千奇百怪 |
| SSE 流 | 重连时排空旧消息 | 防止客户端收到过期数据 |

### 2.2 熔断器四状态模型

```
CLOSED（正常）→ 连续 3 次失败 → OPEN（快速失败）
                                    ↓ 30s 后
                              HALF_OPEN（试探 1 请求）
                                    ↓ 成功
                              CLOSED（恢复）
```

**关键细节**：HALF_OPEN 需要额外的 PROBING 状态 + `asyncio.Lock` 原子转换，否则并发请求会触发"惊群效应"。

### 2.3 回退链设计模式

以 RAG 检索为例的三级回退：

1. **预计算向量**（关键词匹配，不加载模型）
2. **运行时 Embedding**（Ollama 支持，完整语义能力）
3. **优雅降级**（返回空结果，不中断流程）

**教训**：预计算不是银弹。关键词匹配无法捕获语义近义（"违约赔偿上限" vs "惩罚性违约金"）。回退链同时提供内存效率和语义能力。

## 三、法律领域 AI 的特殊铁律

### 3.1 信任是二值的

> 一次漏检或误报 = 系统可信度归零。

这意味着：
- **不存在"大体可用"**——要么可信赖，要么不能用
- **异常路径代码量虽小，影响却最大**（Commit 2 仅 +352 行 ≈10%，却定义了系统可靠性）
- **LLM 说"没问题"不等于没问题**——LLM 漏检不能证明无风险

### 3.2 可审计性 > 便利性

- 每次 LLM 调用记录：维度、分块索引、token 数、缓存命中
- 审计日志用 append-only JSONL 格式（不可篡改）
- 报告包含汇总统计（总调用数、总 token、缓存命中率）
- **"系统为什么这样判断？"必须始终可回答**

### 3.3 独立验证三部曲

> 原则："实施者的自我报告不可信"（来自 superpowers 框架）

1. **证据验证**：检查 `clause_text` 是否存在于原始合同（精确匹配 → 滑动窗口模糊匹配，阈值 0.75）
2. **跨维度关联**：多个维度标记同一条款时提升严重性
3. **矛盾检测**：合规性说"合规"但风险分析说"高风险"时报警

**纯逻辑实现，不需要额外 LLM 调用。**

## 四、开发过程最佳实践

### 4.1 有机生长 + 框架审视 循环

项目验证了一个高效的迭代模式：

```
实践驱动（解决眼前问题）→ 框架审视（系统性发现盲区）→ 规范化（固化为标准）
```

- Commits 1-3：有机生长，解决实际部署中遇到的问题
- Commits 4-5：引入 harness-books/compound-engineering/superpowers 三框架做间隙分析
- Commits 6-8：工具统一、CLAUDE.md 规范化、CI/CD 流水线

**教训**：纯有机生长会遗漏系统性问题（如熔断器、审计追踪）；纯框架驱动会过度设计。两者交替最有效。

### 4.2 大改动后必须有专项质量回合

- Commit 4（863 行）引入了竞态条件、时序攻击漏洞、性能陷阱
- Commit 5（163 行）专门修复上述问题
- **原则**：大特性 PR 后紧跟一个质量稳固 PR

### 4.3 制度化学习：系统越用越好

```
审核完成 → 提取候选规则（LLM 发现 + YAML 中没有 + 严重性达标）
         → 自动生成 AUTO_ 前缀规则 ID
         → 规则版本追踪（SHA-256 哈希 + 时间戳）
```

**价值**：法律经验复利效应——每次审核不只是消耗，还是知识积累。

### 4.4 代码审计驱动改进

项目的 10 项审计清单值得复用：

- [ ] 静默吞异常（`except: pass`）
- [ ] 硬编码密钥/token
- [ ] 魔法数字
- [ ] 死代码/注释掉的废弃代码
- [ ] 异常缺少消息
- [ ] 哈希一致性
- [ ] 重复逻辑
- [ ] Import 组织
- [ ] 类型标注完整性
- [ ] 未使用的 import

## 五、关键反模式速查

| 反模式 | 项目中的真实案例 | 正确做法 |
|--------|----------------|---------|
| 假设部署环境 | 以为"本地"=GPU 服务器 | 先调研真实用户的硬件 |
| 预计算作为默认 | 预计算 embedding 丢失语义能力 | 回退链兼顾效率和能力 |
| 先上线后加固 | 法律领域信任归零不可逆 | 防御式编程从第一天开始 |
| 串行处理 | Embedding 串行批处理 | `asyncio.gather` + `Semaphore` 并发 |
| MD5 哈希 | ingestor.py 用弱哈希 | 统一使用 SHA-256 |
| 实例变量存并发状态 | 审计日志竞态条件 | 函数级局部变量 + 合并 |
| 只存 memory 不入库 | 关键决策仅在 memory 中 | 写入 `docs/` 随 git 提交 |
| 验证脚本用完即弃 | 手动验证不可回归 | 沉淀到测试用例 |

## 六、团队定位：混合型智慧

项目总结了三种 AI 辅助开发范式：

| 范式 | 特点 | 适用场景 |
|------|------|---------|
| Claude Code 式 | 灵活适应，就地沉淀经验 | 探索阶段、原型验证 |
| Codex 式 | 系统化、可复现、易审计 | 生产阶段、合规要求高 |
| **混合式（推荐）** | 明确边界：必须系统化的 vs 可灵活处理的 | 法律 AI 等高信任域 |

**必须系统化的**：规则优先级、审计追踪、验证门禁、经验格式
**可以灵活的**：回退策略、模型选择、分块大小、缓存策略

## 八、已知失败模式速查（调试加速器）

> 来源：tw93/Waza Hunt 技能，从 300+ 实战会话中提炼的 7 种常见故障形状。
> 遇到 bug 时先对照此表快速定位方向，避免无假设的随机试探。

| 失败形状 | 线索特征 | 优先检查位置 |
|---------|---------|------------|
| **时序问题** | 间歇性出现，与负载/并发相关 | 共享状态的并发访问、锁缺失 |
| **缺失守卫** | 字段/索引访问崩溃，空值异常 | 可选值未检查、空集合未处理 |
| **排序 bug** | 单独运行正常，组合执行失败 | 事件回调顺序、事务作用域 |
| **边界失败** | 超时、错误响应格式 | 外部 API 返回值、服务边缘条件 |
| **环境不匹配** | 本地通过但 CI 失败 | 环境变量差异、特性标志、依赖版本 |
| **陈旧值** | 显示旧数据，重启后恢复 | 内存缓存、memoize 结果、缓存 key |
| **状态等待** | 用 `sleep` 固定时间来"修复" | 改为轮询（文件存在？端口响应？进程就绪？） |

**假设验证纪律**：
1. 陈述一个具体假设（精确到文件/函数/行号）
2. 加一个目标探针（日志行、断言、最小测试）
3. 运行验证
4. 证据矛盾 → 彻底丢弃该假设，不留"也许部分对"
5. 连续 3 个假设失败 → 停止，向用户上报

## 九、核心方法论一句话总结

1. **约束即创意**——最好的设计来自真实限制，不是想象中的需求
2. **错误路径即主路径**——防御式编程不是偏执，是工程素养
3. **有机生长 + 框架审视交替**——既不盲目摸索，也不过度设计
4. **第二次出现时统一**——抽象不早不晚
5. **大改动后必须质量回合**——速度的代价是债务，质量的代价是时间，选后者
6. **制度化学习**——系统每次使用都应该变得更好
7. **信任是二值的**——在高风险领域，"大体可用"等于不能用
8. **验证是证据，不是信心**——"应该可以"是理由化，"验证通过"才是完成
