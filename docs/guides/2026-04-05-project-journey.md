# 项目历程：从空仓库到合同审核系统

> AI 合同审核系统 (Contract Reviewer) 完整开发回顾
> 2026-03 → 2026-04 | 10 commits, ~53 files, ~4100 lines

---

## 一、起点：一个空仓库和一个问题

2026 年 3 月 11 日，仓库创建。只有一个 `README.md`，一行字。

问题很具体：**中国企业的合同审查，能不能用 AI 做？** 不是概念验证，是要能跑起来、能落地的东西。这意味着几个硬约束从一开始就存在：

1. **中文法律为主，多法域并存** — 核心场景是中国法律（《民法典》合同编、中文条款结构"第一条"/"（一）"），但系统从一开始就留了国际化钩子：条款解析器已支持英文模式（`Article\s+\d+`、`Section\s+\d+`），规则文件有 `description_en` 双语字段，system prompt 会根据合同语言自适应（"请使用与合同相同的语言回复"）。不过，深层基础设施 — 插件（`CivilCodeCompliancePlugin`）、预计算查询模板、Jinja2 提示词主体 — 仍然是中文特化的。真正的多语言支持是后面的欠账项
2. **部署灵活性** — 有的客户能用云端 API，有的客户的合同不能出内网
3. **法务人员可参与** — 规则不能写死在代码里，法务要能改

## 二、第一刀：系统骨架 (commit e9c00ef)

> +2666 行，46 个文件。从零到一个完整可运行的系统。

### 做了什么决定

**LLM 抽象层选择 LiteLLM。** 这个决定贯穿全项目。一行 `model="anthropic/claude-sonnet-4-20250514"` 是云端，换成 `model="ollama/qwen2.5:72b"` 就是本地。不需要两套代码，不需要 if-else 判断部署模式。环境变量一改，运维搞定。

为什么不选别的方案？简单对比一下：

| 方案 | 优势 | 代价 |
|------|------|------|
| 直连 Ollama SDK | 零额外依赖 | 只覆盖本地模型，云端需要单独接 OpenAI/Anthropic SDK，至少 200 行适配代码 |
| 直连 OpenAI SDK | 生态最大 | Ollama 虽然兼容 OpenAI 格式，但 Anthropic API 格式不同（tool_use 参数结构、错误类型），需要自建适配层 |
| 自建抽象层 | 完全可控 | 300-600 行 provider adapter + 异常映射 + 重试逻辑，后续每接一个新 provider 就要维护一套 |

LiteLLM 加了一个依赖（~50MB），但统一了 `acompletion()`/`aembedding()` 异步 API、模型字符串路由（`anthropic/`、`ollama/`、`openai/` 前缀）、异常类型层级（`RateLimitError`、`APIConnectionError`、`Timeout`）、以及 `api_base` 覆盖本地端点。项目实际用到的就是这些 — 没用它的路由/负载均衡、failover 链或内置缓存（缓存是自己写的 `ResponseCache`，因为需要按模型 ID 区分缓存键）。换句话说，LiteLLM 在这个项目里的角色是"provider 适配层"，不是"AI 网关"。

**RAG 而非微调。** 法律知识用向量检索注入（ChromaDB + bge-large-zh），不做模型微调。原因：
- 法律条文更新频繁（司法解释一年出好几个），微调跟不上
- 向量库可以增量更新，`contract-review ingest` 一条命令
- 法务人员能理解"往知识库里加文档"，不能理解"重新训练模型"

**条款感知分块。** 不用固定 512 token 硬切，而是识别中文合同的章节结构（"第X条"、"一、"、"（一）"正则匹配），按条款边界分块。这样每个 chunk 是一个完整的法律条款，LLM 审查时不会断章取义。

**四维度并发审查。** 风险识别、合规检查、完整性检测、条款公平性 — 四个维度并行发给 LLM，`asyncio.gather` 收结果，最后聚合打分。不是串行等四轮，而是一轮并发出去。

**插件系统。** `plugins/` 目录下的 `builtin/` 有三个内置插件（民法典合规、风险条款、条款公平性），但架构是开放的 — 注册一个新 class 就是一个新审查维度。

**Jinja2 提示词模板 + YAML 规则。** `config/prompts/` 下是 `.jinja2` 模板，`config/rules/default_rules.yaml` 是合规规则。法务人员不需要碰 Python 代码，打开 YAML 文件就能添加规则：

```yaml
- id: "LIABILITY_001"
  description: "违约责任应当对等，不应仅约束一方"
  severity: "medium"
```

**双入口。** CLI（Typer + Rich）给命令行用户，FastAPI + SSE 给前端对接。同一套引擎，两种使用方式。

### 没做什么

没有写测试。没有做 CI。没有做用户认证。这些都是重要的，但第一步是验证核心流程能跑通：PDF → 分块 → RAG 检索 → LLM 审查 → 报告输出。

## 三、第二刀：生产化修复 (commit 6dc718c)

> +352/-136 行，8 个文件。全是 bug fix 和防御性编程。

系统骨架搭完后，开始对每一层做"如果出错会怎样"的审视。发现了一批 P0 问题：

### LLM 层

- **API 调用无 timeout。** LiteLLM 调用卡住了就永远卡住。加了 120 秒超时。
- **Token budget 泄漏。** 并发请求时，如果某个请求异常退出，预算槽位不归还，最终所有请求都在等。改成 `async with` 上下文管理器，保证异常也归还。
- **缓存键不含模型 ID。** 换了模型但命中旧缓存，返回了错误模型的结果。

### 文档解析层

- **扫描件 PDF 无提示。** 扫描件提取出空文本，然后 LLM 收到空字符串开始"幻觉"。加了最小文本长度检查（50 字符），扫描件直接报错提示。
- **编码问题。** 很多中国企业的 `.txt` 合同是 GB18030/GBK 编码，直接 UTF-8 读取就崩。加了编码回退链：UTF-8 → GB18030 → GBK → Latin-1。

### 审查引擎

- **维度索引错位。** 并发审查完成后，结果和维度的对应关系错了。根因：`zip` 用了 `active_dims` 而不是实际运行的 `run_dims`。这种 bug 不写测试很难发现，但审查报告里"风险分析"的内容出现在"合规检查"下面，用户一眼就能看出来。

### 结果聚合

- **去重算法有缺陷。** 原来用字符集合重叠判断两个风险发现是否重复，但"第一条违约金过高"和"第一条付款期限不明"字符重叠也很大。改用 `SequenceMatcher`，阈值 0.7。

### API 层

- **SSE 流不排空。** 客户端断开时，进度事件队列里还有消息没发，下次连接可能收到旧消息。加了 drain 逻辑。

**这个 commit 的核心教训：** 第一版写的是"正常路径"，第二版补的是"异常路径"。异常路径的代码量不多（+352 行），但对可靠性的影响远大于功能代码。

## 四、第三刀：边缘部署能力 (commit dca9190)

> +749/-18 行，13 个文件。两个方向：OCR 扫描件支持 + RAG 内存优化。

### 背景：16GB 笔记本的挑战

"本地部署"不只是在服务器上用 Ollama 跑一个 72B 模型。真实场景是：法务人员的笔记本，16GB 内存，没有独显，手边有一堆扫描件合同。

三个问题浮出水面：

1. **扫描件怎么办？** 第二个 commit 虽然加了扫描件检测，但只是报错。不解决问题。
2. **三个模型同时驻内存？** LLM（~4GB） + Embedding（~0.6GB） + OCR（~1GB） = 至少 5.6GB 模型内存。加上系统和应用开销，16GB 笔记本撑不住。
3. **离线场景？** 有些客户的审核电脑不联网。

### OCR 模块设计

做了技术选型对比后，选择 **PaddleOCR-VL-1.5** 作为默认引擎：

- 参数量只有 0.9B，普通笔记本 CPU 就能跑
- 在 OmniDocBench 上综合表现最好，尤其是中文合同里常见的表格、多栏排版
- 响应时间稳定（和 GLM-OCR 的 100 秒+ 不稳定时间形成对比）
- 手写体识别比 GLM-OCR 好 5 倍 — 很多合同有手写批注

但也保留了 **GLM-OCR** 作为备选，通过工厂模式一行配置切换：

```python
ocr_engine = create_ocr_engine(settings)  # 根据 CR_OCR_PROVIDER 自动选择
```

**关键设计：分阶段加载。**

```
阶段1: 加载 OCR → 识别文字 → 卸载 OCR
阶段2: RAG 检索（用预计算向量，不加载模型）
阶段3: 加载 LLM → 审查 → 完成
```

任何时刻内存中只有一个大模型。峰值内存 = max(单模型大小)，而非 sum(所有模型)。

`PaddleOCREngine.unload()` 不是摆设 — `parse_async` 完成后立即调用，释放模型内存给后续 LLM 使用。

### 预计算查询向量

这是内存优化的另一半。原来的 RAG 流程：

```
query text → [Embedding Model] → vector → ChromaDB → results
```

问题：Embedding 模型（bge-large-zh, ~0.6GB）需要常驻内存。

观察到：合同审查的高频查询模式是可预测的。四个审查维度（风险、合规、完整性、公平性）覆盖了大部分日常场景，每个维度关注的核心法律概念也是已知的。但这不意味着查询空间是封闭的 — 新的审查维度（如数据合规、ESG 条款）随时可能出现，长尾查询（如"跨境数据传输限制条款"）无法被 16 个预定义模板覆盖。

所以改成：

```
[离线] query templates → [Embedding Model] → vectors → precomputed_queries.json
[运行时] query text → keyword match → precomputed vector → ChromaDB → results
```

16 个预定义的法律概念模板，覆盖四个审查维度。运行时用关键词匹配找到最接近的模板，用它的预计算向量去 ChromaDB 查。纯预计算模式下不需要加载 Embedding 模型，0.6GB 内存可以让出来。

`scripts/precompute_queries.py` 在有 Embedding 模型的机器上运行一次，生成的 JSON 文件随 ChromaDB 向量库一起分发到边缘节点。

### 预计算的局限性（commit 9f9c299 自我修正）

回头看，"去掉 Embedding 模型"这个表述过于绝对了。预计算模式适合纯离线、极端内存受限的边缘部署场景（比如 16GB 笔记本同时跑 OCR + LLM 时确实没余量），但作为系统默认行为，保留 bge-large-zh 通过 Ollama 常驻后台是更好的选择。理由有三：

1. **语义检索能力不受模板限制。** 预计算的 keyword 匹配是简单的子字符串 `in` 检查（`precomputed_queries.py:98`），无法捕捉语义近义。"违约赔偿上限"和"惩罚性违约金"在语义上高度相关，keyword 匹配发现不了。
2. **长尾查询有去处。** 16 个模板覆盖了高频概念，但法务人员提出"数据跨境传输""竞业限制期限"这类新维度时，纯预计算模式的 retrieval 直接返回空。Runtime embedding 能给出合理结果。
3. **新维度上线零成本。** 添加新审查维度时不需要重新运行 `precompute_queries.py` 并重新分发 JSON 文件。

实际上，`retriever.py:87-102` 已经实现了 fallback 链（precomputed → runtime embedding），只是原来 `cli.py` 在预计算模式下把 embedder 设为 None，fallback 形同虚设。修复后的默认模式是 `runtime_embed`：Ollama 跑着就用实时向量，Ollama 没启动则 gracefully 降级到预计算。

### 分布式预留

`OCRWorker` 里有一个 `result_endpoint` 参数和 `_submit_result` 方法。当前版本里这个是可选的，但架构上是为 Layer 3（中央聚合）预留的：

- 合同原件留在边缘设备（安全）
- OCR 提取的纯文本可以（可选地）提交到中央服务
- 提交失败不影响本地审查流程（non-fatal）

---

三个 commit 下来，系统从骨架变成了能处理扫描件、能在笔记本上跑的产品雏形。但进化方式一直是"碰到问题补一个" — 扫描件不能处理就加 OCR，内存不够就做预计算，编码出错就加 fallback。每一个修补都是对的，但我不知道还有什么没补到。需要一个框架来系统性审视自己。

## 五、停下来看看：Harness Engineering 框架审视

> 基于 harness-books、compound-engineering、superpowers 三个开源项目的对照分析。

### 三个参照系

回头看我们的系统，它已经是一个 agent 系统了 — 虽然我们从没这样叫它。LLM 调用有 timeout 和 token 预算，四个审查维度并发跑，RAG 有 fallback 链，OCR 分阶段加载释放内存。这些就是 agent 系统的运行时基础设施。

问题是：我们是"碰到问题补一个"地长出来的，没有一个系统性的框架来审视哪些做了、哪些漏了。三个开源项目提供了这样的框架：

- **harness-books** 给出了理论骨架 — 六个维度（控制面、连续性、权限、本地治理、错误恢复、多代理验证），每个维度有明确的"达标线"
- **compound-engineering** 给出了工作流实践 — 多 persona 并行审查、结果流水线、机构知识沉淀
- **superpowers** 给出了验证纪律 — "声明完成前必须有证据"、"实施者的自我报告不可信"、反合理化护栏

### 做到了什么

逐维度对照后，发现有些地方做得比预期好：

- **资源生命周期管理** — 分阶段加载保证峰值内存 = max(单模型)，token 预算 + Semaphore 并发控制防止 API 失控
- **三层 Prompt 分离** — Jinja2 模板（系统级）、YAML 规则（业务级）、环境变量（运维级），三层的修改者和修改频率不同
- **维度级异常隔离** — `asyncio.gather` 的 `return_exceptions=True` 保证一个维度崩了不影响其他维度
- **防御性 JSON 解析** — 三层防御（tool_calls → json.loads → raw_response），模型返回格式不对也不崩

### 六个维度，六处缺口

| 维度 | 核心缺口 | 影响 |
|------|----------|------|
| **控制面** | 无优先级机制 — YAML 规则与 Jinja2 模板冲突时无消解；无 prompt 质量门控 | 残缺 prompt 可能发给 LLM |
| **连续性** | 无上下文窗口预检 — 长 chunk + 系统 prompt 可能超出模型限制 | API 返回错误才发现 |
| **权限** | 无 API 认证、无文件路径校验、无请求配额 | API 裸跑，存在安全隐患 |
| **本地治理** | 无机构记忆 — 审查完一份合同就结束，不记得上一份发现了什么；规则无版本追踪 | 系统不会越用越好 |
| **错误恢复** | 无指数退避重试、无熔断器 — Ollama 挂了所有维度各自超时 120 秒 | 网络抖动导致审查失败 |
| **多代理验证** | 维度间无协调、无独立验证阶段、无 audit trail | LLM 幻觉无法被发现 |

### 验证铁律的法务映射

superpowers 项目的"反合理化护栏"在法务场景下意义被放大了 — 合同审核的后果比代码审查严重得多：

| 软件开发中的借口 | 法务审查中的对应 | 为什么危险 |
|----------------|----------------|----------|
| "太简单不需要测试" | "这个条款是标准的，不用审" | 标准条款被修改一个数字就变成不标准 |
| "测试通过了所以代码是对的" | "LLM 说没问题所以没问题" | LLM 漏检不代表没有风险 |
| "我之后补测试" | "先出报告，有问题后面修" | 报告出了就会被用于决策，没有"后面" |
| "改动太小不会有影响" | "只是措辞变化，实质不变" | 法律文本的措辞就是实质 |

### 三件事，按优先级

如果只能改三件事：

1. **P0：指数退避 + 熔断器** — 最低成本、最高确定性。`LLMClient.complete()` 对 429/5xx 重试 3 次，间隔 1s/2s/4s；新增 `CircuitBreaker`：连续 3 次失败 → 打开（fail-fast）→ 30 秒后半开（试一次）→ 成功则关闭
2. **P1：独立验证阶段** — 四个维度跑完后，验证每个发现是否有原文依据、不同维度是否矛盾、多维度同时标记的问题提升 severity。纯逻辑验证，不需要额外 LLM 调用
3. **P2：审查经验沉淀** — 审查完成后，将有价值的发现沉淀为规则条目，下次审查同类合同时自动检索

分析完了，该动手了。从 P0 开始。

## 六、第四刀：五阶段 Harness Engineering 落地 (commit 123f0e4)

> +863/-32 行，16 个文件。这是项目体量最大的一次改动。

第五章的分析给出了方向，这个 commit 把方向变成代码。五个阶段对应五个能力层：

### Phase 1：韧性层 — 让系统能扛住抖动

对应第五章的 P0 缺口"无重试、无熔断"。

新增 `llm/retry.py`（51 行）— 指数退避重试工具。可配置最大重试次数、基础延迟、抖动。只对特定异常类型重试（限流 429、服务端 5xx），4xx 请求错误直接失败。

新增 `llm/circuit_breaker.py`（114 行）— 四状态熔断器（CLOSED → OPEN → HALF_OPEN → PROBING）。连续失败达到阈值后打开熔断器，所有后续调用直接 fail-fast，不再浪费时间等超时。30 秒后进入 HALF_OPEN，放一个探测请求过去；成功则关闭，失败则重新打开。

关键设计：`coro_factory()` 模式 — 熔断器接收的不是协程本身，而是创建协程的工厂函数。这样在 OPEN 状态下不会创建不必要的 API 调用对象。

`LLMClient.complete()` 和 `Embedder._embed_batch()` 都接入了 retry + circuit breaker 双重保护。

### Phase 2：独立验证 — 不信任 LLM 的自我报告

对应第五章的 P1 缺口"无独立验证阶段"。

新增 `review/verifier.py`（211 行）— 审查结果验证器，三重验证：

1. **证据核实** — 检查每个 RiskFinding 的 `clause_text` 是否在合同原文中存在。先精确匹配，再滑动窗口模糊匹配（SequenceMatcher，阈值 0.75）。找不到证据的发现自动降级为 INFO 并标注"缺乏原文依据"
2. **交叉印证** — 如果同一段条款被两个以上维度同时标记，自动提升 severity。O(n²) 两两比较，对于典型的 <50 个发现完全可接受
3. **矛盾检测** — 如果 compliance 维度说"合规"但 risk_analysis 说"高风险"，标记矛盾

这是 superpowers "实施者的自我报告不可信"原则的直接落地。

### Phase 3：审计追踪 — 让每个决定可回溯

对应第五章"无 audit trail"缺口。

新增 `review/audit.py`（75 行）— 不可变审计日志。每个 LLM 调用记录维度、chunk 索引、token 消耗、是否命中缓存。写入 JSONL 格式，一行一事件，`append-only`。审查报告附带摘要统计（总调用数、总 token、缓存命中率）。

法务问"这个结论怎么来的"，现在有据可查。

### Phase 4：机构记忆 — 让系统越用越好

对应第五章的 P2 缺口"无审查经验沉淀"。

新增 `review/learnings.py`（96 行）— 从审查结果中提取候选规则。逻辑：如果某个 RiskFinding 不属于已有 YAML 规则（通过模糊匹配判断，阈值 0.5），且 severity 达标，则生成候选规则条目（自动 ID 前缀 `AUTO_`）。按风险类型推断分类（liability、penalty、termination 等）。

新增 `review/rule_history.py`（55 行）— 规则文件版本追踪。每次规则变更记录 SHA-256 指纹、时间戳、文件大小到 `rule_versions.jsonl`。解决"法务改了规则引入误报，没法回溯"的问题。

规则 YAML 格式也扩展了 `added_date` 和 `source` 字段（`manual` / `auto_discovered`）。

### Phase 5：可扩展性 — 为外部集成预留钩子

新增 `review/hooks.py`（72 行）— Protocol-based 生命周期钩子。三个挂载点：`on_review_start`、`on_dimension_complete`、`on_report_ready`。异常隔离 — 钩子崩了不影响审查流程，日志记录但不传播。

API 层加了基础的 API key 认证（`X-API-Key` header），解决第五章"API 裸跑"的安全缺口。

## 七、第五刀：质量修复 (commit c3b78bd)

> +163/-79 行，10 个文件。大改之后的稳定轮。

第六章的 863 行代码引入了几类问题，需要专门的质量修复：

### 并发安全

- **审计日志竞态。** `audit.py` 原来用实例变量存事件列表，多个维度并发写入时有竞态。改为函数级局部变量 + 最终合并
- **熔断器 thundering herd。** HALF_OPEN 状态下多个请求同时到达，都认为自己是"探测请求"。新增 PROBING 状态 — 只有第一个请求进入探测，其余直接拒绝。用 `asyncio.Lock` 保证状态转换原子性

### 代码整洁

- **重试逻辑重复。** `LLMClient` 和 `Embedder` 各自实现了一套重试。提取为共享的 `llm/retry.py`，两处复用同一个 `retry_with_backoff()`
- **Pydantic Field 修正。** 部分字段用了 Python 默认值而非 `Field(default=...)`，Pydantic v2 会 warn
- **API key 比较。** 原来用 `==` 比较 API key，存在时序攻击风险。改为 `hmac.compare_digest()`

### 验证器优化

- **大小写 fast path。** `clause_text` 大小写不敏感预比较，匹配成功直接返回 1.0，跳过 SequenceMatcher 开销
- **滑动窗口步长。** 原来逐字符滑动，改为四分之一窗口步长，在精度可接受的范围内减少约 75% 的比较次数

**教训：大改之后必须有一轮专门的质量审查。** 863 行里藏着竞态条件、时序攻击、性能陷阱 — 这些不是功能 bug，正常测试不一定能覆盖，需要专门带着"这段代码有什么隐患"的视角去审。

## 八、第六刀：工具统一与可观测性 (commit c7e6cf4)

> +111/-48 行，13 个文件。消除技术债 + 增强运维能力。

### 工具统一

审计过程中发现两类重复实现：

- **哈希计算** — `ingestor.py`、`rule_history.py`、`audit.py` 三个文件各自实现了文件哈希。而且 `ingestor.py` 用的是 MD5（弱哈希）。统一提取到 `utils/hashing.py`（20 行），全部改为 SHA-256
- **JSONL 写入** — `audit.py` 和 `rule_history.py` 各自实现了 JSONL append。提取到 `utils/jsonl.py`（22 行）

**教训：工具统一应该在第二次出现重复时就做，不是第三次。**

### 可观测性

- **健康端点扩展。** `/health` 端点现在暴露熔断器状态（CLOSED/OPEN/HALF_OPEN/PROBING）和 token 预算使用率。运维不用看日志就能判断系统是否健康

## 十二、回头看的体会

**"先跑通再跑好"是对的，但"跑好"要紧跟着来。** commit 1 和 commit 2 之间隔了不到一天，但 commit 2 修的问题（budget 泄漏、维度错位、编码崩溃）如果到了用户手里才发现，信任就没了。合同审查不是社交软件，错一次法务就不会再用了。

**内存是第一公民。** 一开始没想到 16GB 笔记本的场景，觉得"本地部署"就是有 GPU 服务器。后来发现真实用户就是笔记本 + 扫描件，被迫做了分阶段加载和预计算向量。这两个优化加起来省了 ~1.6GB 模型内存，是让"能跑"和"跑不动"的分界线。

**配置即接口。** YAML 规则、Jinja2 模板、环境变量配置 — 这些不是"锦上添花"，是让非工程师能参与系统演进的接口。法务加一条规则、运维换一个模型、产品改一段提示词，都不需要动 Python 代码。

**异常路径的代码量比你想象的少，影响比你想象的大。** commit 2 只有 +352 行（全项目的 ~10%），但修的都是"系统在真实环境里会不会崩"的问题。

**有机生长和框架审视是互补的。** 前三个 commit 是有机生长 — 碰到问题补一个。第五章的框架审视让我看到了没碰到但一定会碰到的问题。两种模式都需要：纯有机生长缺乏系统性，纯框架驱动缺乏对真实场景的敏感。最好的节奏是"长一阵，停下来看看，再长一阵"。

**法务场景对"可审计"的要求比软件开发高得多。** 一个代码审查工具漏了一个 style issue，没人追责。一个合同审查工具漏了一个风险条款，可能要上法庭解释"你们系统为什么没审出来"。这决定了审计追踪、证据核实、版本追踪不是可选项，是必须项。

**大改之后必须有质量轮。** commit 4 的 863 行里藏着竞态条件、时序攻击、性能陷阱。这些不是功能 bug，正常路径测试发现不了，需要专门带着"这段代码有什么隐患"的视角去审。如果没有 commit 5 的质量修复，这些问题会在生产环境里慢慢暴露。

**规范不是约束，是杠杆。** CLAUDE.md 不是写来限制开发的，是把八个 commit 里学到的经验显式化。下一次开发（不管是谁来做）不需要重新踩这些坑。规范的价值不在于"遵守"，在于"少走弯路"。
