# 项目历程：从空仓库到合同审核系统

> AI 合同审核系统 (Contract Reviewer) 完整开发回顾
> 2026-03 → 2026-04 | 10 commits, ~53 files, ~4100 lines

---

## 一、起点：一个空仓库和一个问题

2026 年 3 月 11 日，仓库创建。只有一个 `README.md`，一行字。

问题很具体：**中国企业的合同审查，能不能用 AI 做？** 不是概念验证，是要能跑起来、能落地的东西。这意味着几个硬约束从一开始就存在：

1. **中文法律为主，多法域并存** — 核心场景是中国法律（《民法典》合同编、中文条款结构"第一条"/"（一）"），但系统从一开始就留了国际化钩子：条款解析器已支持英文模式（`Article\s+\d+`、`Section\s+\d+`），规则文件有 `description_en` 双语字段，system prompt 会根据合同语言自适应（"请使用与合同相同的语言回复"）。不过，深层基础设施 — 插件（`CivilCodeCompliancePlugin`）、预计算查询模板、Jinja2 提示词主体 — 仍然是中文特化的。真正的多语言支持是后面的欠账项
2. **部署灵活性** — 有的客户能用云端 API，有的客户的合同不能出内网
3. **法务人员可参与** — 规则不能写死在代码里，法务要能改

## 二、第一刀：系统骨架 (commit e9c00ef)

> +2666 行，46 个文件。从零到一个完整可运行的系统。

### 做了什么决定

**LLM 抽象层选择 LiteLLM。** 这个决定贯穿全项目。一行 `model="anthropic/claude-sonnet-4-20250514"` 是云端，换成 `model="ollama/qwen2.5:72b"` 就是本地。不需要两套代码，不需要 if-else 判断部署模式。环境变量一改，运维搞定。

为什么不选别的方案？简单对比一下：

| 方案 | 优势 | 代价 |
|------|------|------|
| 直连 Ollama SDK | 零额外依赖 | 只覆盖本地模型，云端需要单独接 OpenAI/Anthropic SDK，至少 200 行适配代码 |
| 直连 OpenAI SDK | 生态最大 | Ollama 虽然兼容 OpenAI 格式，但 Anthropic API 格式不同（tool_use 参数结构、错误类型），需要自建适配层 |
| 自建抽象层 | 完全可控 | 300-600 行 provider adapter + 异常映射 + 重试逻辑，后续每接一个新 provider 就要维护一套 |

LiteLLM 加了一个依赖（~50MB），但统一了 `acompletion()`/`aembedding()` 异步 API、模型字符串路由（`anthropic/`、`ollama/`、`openai/` 前缀）、异常类型层级（`RateLimitError`、`APIConnectionError`、`Timeout`）、以及 `api_base` 覆盖本地端点。项目实际用到的就是这些 — 没用它的路由/负载均衡、failover 链或内置缓存（缓存是自己写的 `ResponseCache`，因为需要按模型 ID 区分缓存键）。换句话说，LiteLLM 在这个项目里的角色是"provider 适配层"，不是"AI 网关"。

**RAG 而非微调。** 法律知识用向量检索注入（ChromaDB + bge-large-zh），不做模型微调。原因：
- 法律条文更新频繁（司法解释一年出好几个），微调跟不上
- 向量库可以增量更新，`contract-review ingest` 一条命令
- 法务人员能理解"往知识库里加文档"，不能理解"重新训练模型"

**条款感知分块。** 不用固定 512 token 硬切，而是识别中文合同的章节结构（"第X条"、"一、"、"（一）"正则匹配），按条款边界分块。这样每个 chunk 是一个完整的法律条款，LLM 审查时不会断章取义。

**四维度并发审查。** 风险识别、合规检查、完整性检测、条款公平性 — 四个维度并行发给 LLM，`asyncio.gather` 收结果，最后聚合打分。不是串行等四轮，而是一轮并发出去。

**插件系统。** `plugins/` 目录下的 `builtin/` 有三个内置插件（民法典合规、风险条款、条款公平性），但架构是开放的 — 注册一个新 class 就是一个新审查维度。

**Jinja2 提示词模板 + YAML 规则。** `config/prompts/` 下是 `.jinja2` 模板，`config/rules/default_rules.yaml` 是合规规则。法务人员不需要碰 Python 代码，打开 YAML 文件就能添加规则：

```yaml
- id: "LIABILITY_001"
  description: "违约责任应当对等，不应仅约束一方"
  severity: "medium"
```

**双入口。** CLI（Typer + Rich）给命令行用户，FastAPI + SSE 给前端对接。同一套引擎，两种使用方式。

### 没做什么

没有写测试。没有做 CI。没有做用户认证。这些都是重要的，但第一步是验证核心流程能跑通：PDF → 分块 → RAG 检索 → LLM 审查 → 报告输出。

## 三、第二刀：生产化修复 (commit 6dc718c)

> +352/-136 行，8 个文件。全是 bug fix 和防御性编程。

系统骨架搭完后，开始对每一层做"如果出错会怎样"的审视。发现了一批 P0 问题：

### LLM 层

- **API 调用无 timeout。** LiteLLM 调用卡住了就永远卡住。加了 120 秒超时。
- **Token budget 泄漏。** 并发请求时，如果某个请求异常退出，预算槽位不归还，最终所有请求都在等。改成 `async with` 上下文管理器，保证异常也归还。
- **缓存键不含模型 ID。** 换了模型但命中旧缓存，返回了错误模型的结果。

### 文档解析层

- **扫描件 PDF 无提示。** 扫描件提取出空文本，然后 LLM 收到空字符串开始"幻觉"。加了最小文本长度检查（50 字符），扫描件直接报错提示。
- **编码问题。** 很多中国企业的 `.txt` 合同是 GB18030/GBK 编码，直接 UTF-8 读取就崩。加了编码回退链：UTF-8 → GB18030 → GBK → Latin-1。

### 审查引擎

- **维度索引错位。** 并发审查完成后，结果和维度的对应关系错了。根因：`zip` 用了 `active_dims` 而不是实际运行的 `run_dims`。这种 bug 不写测试很难发现，但审查报告里"风险分析"的内容出现在"合规检查"下面，用户一眼就能看出来。

### 结果聚合

- **去重算法有缺陷。** 原来用字符集合重叠判断两个风险发现是否重复，但"第一条违约金过高"和"第一条付款期限不明"字符重叠也很大。改用 `SequenceMatcher`，阈值 0.7。

### API 层

- **SSE 流不排空。** 客户端断开时，进度事件队列里还有消息没发，下次连接可能收到旧消息。加了 drain 逻辑。

**这个 commit 的核心教训：** 第一版写的是"正常路径"，第二版补的是"异常路径"。异常路径的代码量不多（+352 行），但对可靠性的影响远大于功能代码。

## 四、第三刀：边缘部署能力 (commit dca9190)

> +749/-18 行，13 个文件。两个方向：OCR 扫描件支持 + RAG 内存优化。

### 背景：16GB 笔记本的挑战

"本地部署"不只是在服务器上用 Ollama 跑一个 72B 模型。真实场景是：法务人员的笔记本，16GB 内存，没有独显，手边有一堆扫描件合同。

三个问题浮出水面：

1. **扫描件怎么办？** 第二个 commit 虽然加了扫描件检测，但只是报错。不解决问题。
2. **三个模型同时驻内存？** LLM（~4GB） + Embedding（~0.6GB） + OCR（~1GB） = 至少 5.6GB 模型内存。加上系统和应用开销，16GB 笔记本撑不住。
3. **离线场景？** 有些客户的审核电脑不联网。

### OCR 模块设计

做了技术选型对比后，选择 **PaddleOCR-VL-1.5** 作为默认引擎：

- 参数量只有 0.9B，普通笔记本 CPU 就能跑
- 在 OmniDocBench 上综合表现最好，尤其是中文合同里常见的表格、多栏排版
- 响应时间稳定（和 GLM-OCR 的 100 秒+ 不稳定时间形成对比）
- 手写体识别比 GLM-OCR 好 5 倍 — 很多合同有手写批注

但也保留了 **GLM-OCR** 作为备选，通过工厂模式一行配置切换：

```python
ocr_engine = create_ocr_engine(settings)  # 根据 CR_OCR_PROVIDER 自动选择
```

**关键设计：分阶段加载。**

```
阶段1: 加载 OCR → 识别文字 → 卸载 OCR
阶段2: RAG 检索（用预计算向量，不加载模型）
阶段3: 加载 LLM → 审查 → 完成
```

任何时刻内存中只有一个大模型。峰值内存 = max(单模型大小)，而非 sum(所有模型)。

`PaddleOCREngine.unload()` 不是摆设 — `parse_async` 完成后立即调用，释放模型内存给后续 LLM 使用。

### 预计算查询向量

这是内存优化的另一半。原来的 RAG 流程：

```
query text → [Embedding Model] → vector → ChromaDB → results
```

问题：Embedding 模型（bge-large-zh, ~0.6GB）需要常驻内存。

观察到：合同审查的高频查询模式是可预测的。四个审查维度（风险、合规、完整性、公平性）覆盖了大部分日常场景，每个维度关注的核心法律概念也是已知的。但这不意味着查询空间是封闭的 — 新的审查维度（如数据合规、ESG 条款）随时可能出现，长尾查询（如"跨境数据传输限制条款"）无法被 16 个预定义模板覆盖。

所以改成：

```
[离线] query templates → [Embedding Model] → vectors → precomputed_queries.json
[运行时] query text → keyword match → precomputed vector → ChromaDB → results
```

16 个预定义的法律概念模板，覆盖四个审查维度。运行时用关键词匹配找到最接近的模板，用它的预计算向量去 ChromaDB 查。纯预计算模式下不需要加载 Embedding 模型，0.6GB 内存可以让出来。

`scripts/precompute_queries.py` 在有 Embedding 模型的机器上运行一次，生成的 JSON 文件随 ChromaDB 向量库一起分发到边缘节点。

### 预计算的局限性（commit 9f9c299 自我修正）

回头看，"去掉 Embedding 模型"这个表述过于绝对了。预计算模式适合纯离线、极端内存受限的边缘部署场景（比如 16GB 笔记本同时跑 OCR + LLM 时确实没余量），但作为系统默认行为，保留 bge-large-zh 通过 Ollama 常驻后台是更好的选择。理由有三：

1. **语义检索能力不受模板限制。** 预计算的 keyword 匹配是简单的子字符串 `in` 检查（`precomputed_queries.py:98`），无法捕捉语义近义。"违约赔偿上限"和"惩罚性违约金"在语义上高度相关，keyword 匹配发现不了。
2. **长尾查询有去处。** 16 个模板覆盖了高频概念，但法务人员提出"数据跨境传输""竞业限制期限"这类新维度时，纯预计算模式的 retrieval 直接返回空。Runtime embedding 能给出合理结果。
3. **新维度上线零成本。** 添加新审查维度时不需要重新运行 `precompute_queries.py` 并重新分发 JSON 文件。

实际上，`retriever.py:87-102` 已经实现了 fallback 链（precomputed → runtime embedding），只是原来 `cli.py` 在预计算模式下把 embedder 设为 None，fallback 形同虚设。修复后的默认模式是 `runtime_embed`：Ollama 跑着就用实时向量，Ollama 没启动则 gracefully 降级到预计算。

### 分布式预留

`OCRWorker` 里有一个 `result_endpoint` 参数和 `_submit_result` 方法。当前版本里这个是可选的，但架构上是为 Layer 3（中央聚合）预留的：

- 合同原件留在边缘设备（安全）
- OCR 提取的纯文本可以（可选地）提交到中央服务
- 提交失败不影响本地审查流程（non-fatal）

---

三个 commit 下来，系统从骨架变成了能处理扫描件、能在笔记本上跑的产品雏形。但进化方式一直是"碰到问题补一个" — 扫描件不能处理就加 OCR，内存不够就做预计算，编码出错就加 fallback。每一个修补都是对的，但我不知道还有什么没补到。需要一个框架来系统性审视自己。
