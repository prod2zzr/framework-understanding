# Harness Engineering 视角下的合同审核系统

> 从 harness-books、compound-engineering、superpowers 三个开源项目看我们的现状与改进方向
> 2026-04 | 基于 wquguru/harness-books, EveryInc/compound-engineering-plugin, obra/superpowers

---

## 一、为什么法务 AI 需要 Harness Engineering

合同审核不是聊天机器人。聊天机器人回错了一句话，用户重新问就行了。合同审核系统漏掉了一个违约金不对等条款，客户签了合同，后果是真金白银。**错一次，法务就不会再用了。**

但回头看我们的系统，它已经是一个 agent 系统了 — 虽然我们从没这样叫它。LLM 调用有 timeout 和 token 预算（`client.py:18,33`），四个审查维度并发跑（`engine.py:86-101`），RAG 有三级 fallback 链（`retriever.py:87-103`），OCR 分阶段加载释放内存（`paddle_engine.py:unload()`）。这些就是 agent 系统的运行时基础设施。

问题是：我们是"碰到问题补一个"地长出来的，没有一个系统性的框架来审视哪些做了、哪些漏了。三个开源项目提供了这样的框架：

- **harness-books** 给出了理论骨架 — 六个维度（控制面、连续性、权限、本地治理、错误恢复、多代理验证），每个维度有明确的"达标线"
- **compound-engineering** 给出了工作流实践 — 多 persona 并行审查、结果流水线、机构知识沉淀
- **superpowers** 给出了验证纪律 — "声明完成前必须有证据"、"实施者的自我报告不可信"、反合理化护栏

以下逐维度对照。每一节的结构是：**开源项目怎么说 → 我们做到了什么 → 我们缺什么 → 可以怎么改。**

## 二、控制面：Prompt 分层与指令治理

### 开源项目怎么说

harness-books 第 2 章的核心论点：**Prompt 不是人格设定，是运行时控制面。** 一个 agent 系统的 prompt 是多个来源（常量、系统指令、记忆、输出格式要求）在运行时动态拼装的结果，需要有明确的优先级和冲突消解规则。Book 2 进一步区分了两种控制面风格 — Claude Code 式的"动态拼装"（灵活但长会话中指令会漂移）和 Codex 式的"结构化片段"（可审计但维护成本高）。

compound-engineering 的 `/ce:plan` 命令在执行前做"置信度检查" — 计划不够好就不开始执行。这是对"控制面质量"的运行时门控。

### 我们做到了什么

三层分离已经存在：

1. **Jinja2 模板**（`config/prompts/system.jinja2`）— 系统级指令，定义审查角色和输出格式
2. **YAML 规则**（`config/rules/default_rules.yaml`）— 业务级约束，法务可编辑
3. **环境变量**（`models/config.py:56` 的 `env_prefix="CR_"`）— 运维级配置，模型选择、API 端点

这三层的修改者不同（开发者、法务、运维），修改频率不同，这是好的。

### 我们缺什么

- **无优先级机制。** 如果 YAML 规则说"违约金超过合同总额 30% 为高风险"，但 Jinja2 模板里的通用指令说"20% 以上即高风险"，谁赢？目前没有冲突消解。
- **无运行时指令注入。** 审查过程中没有动态调整指令的能力。比如第一轮审查发现合同是金融行业，第二轮应该自动注入金融监管相关指令 — 目前做不到。
- **无 prompt 质量门控。** compound-engineering 的置信度检查思路值得借鉴：prompt 组装完成后，在发给 LLM 之前，先检查是否完整（规则是否加载了？上下文 chunk 是否为空？模板渲染是否成功？）。

### 可落地的改进

在 `PromptBuilder.build_review_prompt()` 里加一个 pre-flight 校验：检查规则列表非空、chunk 文本长度合理、模板渲染结果包含必要的关键段落。校验不通过就 early return 带明确错误信息，不要把一个残缺的 prompt 发给 LLM。

## 三、连续性：上下文治理与资源生命周期

### 开源项目怎么说

harness-books 第 5 章："**上下文是工作记忆，治理的目标是让系统能继续工作。**" 具体做法包括 MEMORY.md 严格限制 200 行 / 25000 字节、session 记忆有 9 字段结构化模板（2000 tokens/段、12000 tokens 总预算）、autoCompact 留 20000 tokens 余量并设置连续失败 3 次的熔断。核心原则：**优先保护行动语义，而非信息完整性。**

superpowers 的 subagent 模式中，协调者给每个 subagent "策划好的上下文"（curated context），subagent 不自己读计划文件。这是对上下文大小的主动管理。

### 我们做到了什么

**资源生命周期管理是我们的强项。**

分阶段加载确保峰值内存 = max(单模型大小)：

```
阶段1: 加载 OCR → 识别文字 → 卸载 OCR（paddle_engine.py:unload()）
阶段2: RAG 检索（precomputed 模式不加载模型，runtime_embed 走 Ollama）
阶段3: 加载 LLM → 审查 → 完成
```

Token 预算追踪（`client.py:33` 的 `TokenBudget`）加 Semaphore 并发控制（`llm_max_concurrent=5`），防止 API 调用失控。每个 chunk 独立处理，不跨 chunk 累积上下文状态。

### 我们缺什么

- **无上下文窗口预检。** `PromptBuilder` 组装完 prompt 后不检查是否超出模型的 `max_tokens`。如果一个 chunk 很长，加上系统 prompt 和规则注入，可能超出模型限制，直到 API 返回错误才发现。
- **无 compact 机制。** 在当前的单轮审查场景下这不是大问题，但如果未来做多轮交互审查（法务提问 → 系统解释 → 法务追问），上下文会快速膨胀，需要压缩策略。
- **prompt 大小不确定。** 我们用 `chunk_size_tokens=512` 控制 chunk 大小，但 RAG 检索结果、规则列表、系统 prompt 的大小没有纳入预算。superpowers 的"策划好的上下文"思路在这里适用：在发给 LLM 前，先算一遍总 token 数，超了就裁剪 RAG 结果或规则列表（按相关性排序后截断）。

## 四、工具与权限：谁来阻止系统动手太快

### 开源项目怎么说

harness-books 第 4 章的核心观点："**一旦模型开始调用工具，问题的性质就变了。**" 文本错误可以重新生成，工具执行产生的是真实后果。关键设计：

- 权限三态模型（allow / deny / ask）— 不是二元的"能不能用"，而是"直接允许 / 直接拒绝 / 暂停并询问"
- **能力与信任反相关** — Bash 能力最大，所以限制最严
- 工具执行前经过权限检查、hooks、遥测、合成错误处理

compound-engineering 的文档审查结果路由更精妙：不是按严重度（P0/P1/P2）分流，而是按"是否存在唯一正确修复"分流 — **有明确修复的自动改，需要判断的交给人。** 这个思路比简单的严重度阈值高明。

### 我们做到了什么

- **Semaphore 并发控制**（`batch.py:35` 的 `asyncio.Semaphore(max_concurrent_contracts)`）— 防止同时处理太多合同压垮系统
- **120 秒超时**（`client.py:18` 的 `DEFAULT_TIMEOUT`）— 防止 LLM 调用无限挂起
- **Settings 启动后基本只读** — 配置不会在运行中被意外修改（CLI 层面的 `settings.ocr_enabled = True` 是个例外）

### 我们缺什么

- **无 API 认证。** FastAPI 层裸跑，在 `project-retrospective.md` 第六章已经承认了这个欠账。
- **无文件路径校验。** CLI 接受用户传入的合同文件路径，没有检查路径遍历（`../../etc/passwd`）。对于只在本地跑的工具这不算严重，但如果部署为 API 服务就是安全漏洞。
- **无请求配额。** 没有 per-client 或 per-时段的调用限制。
- **审查结果只有严重度，没有"可自动处理性"维度。** compound-engineering 的 auto-fix / present 分流启发我们思考：有些审查发现（如"缺少争议解决条款"）有明确的模板化建议，有些（如"违约金比例是否合理"）需要法务判断。当前的 severity（HIGH/MEDIUM/LOW/INFO）没有区分这两类。

## 五、本地治理：规则沉淀与机构记忆

### 开源项目怎么说

harness-books Book 2 第 5 章对比了两种治理风格：

- **Claude Code 式 — 现场经验沉淀。** CLAUDE.md、skills、hooks、session memory 组合成"到哪个项目就吸收哪个项目规则"的能力。像一个"走到哪都带着笔记本的工程师"。优点：适应快。缺点：各地规则散落，没有统一审计。
- **Codex 式 — 制度注入。** Skills 是有版本、有指纹追踪的安装资产，AGENTS.md 有层级作用域。像一个"先贴规章再干活的项目经理"。优点：可复制、可审计。缺点：学习成本高。

compound-engineering 的 `/ce:compound` 命令是"复利工程"的核心 — 每次解决问题后，将方案写成结构化文档（YAML frontmatter 标注 module、problem_type、root_cause、severity），存入分类目录（`docs/solutions/performance-issues/`、`docs/solutions/security-issues/` 等）。下次开始新工作前，`learnings-researcher` agent 自动检索已有方案：Grep 预筛 → 读 frontmatter → 评分排序 → 只深读相关文档，目标 30 秒内返回。

superpowers 把每个能力封装成 skill（`writing-plans.md`、`test-driven-development.md`、`systematic-debugging.md`），每个 skill 有明确的触发条件、步骤定义和退出标准。Skills 是可组合的积木，不是散落的经验。

### 我们做到了什么

- **YAML 规则可编辑** — 法务打开 `default_rules.yaml` 就能添加合规规则，不需要碰 Python 代码
- **环境变量配置** — 运维通过 `CR_*` 环境变量切换模型、API 端点、RAG 模式
- **Jinja2 模板** — 提示词修改不需要重新部署代码

### 我们缺什么

**机构记忆是最大的缺口。**

目前的系统审查完一份合同就结束了。它不记得上一份合同发现了什么。如果法务在审查报告里标注"这个风险判断是错的"，这个反馈无处可去 — 不会改进下一次审查。

具体缺失：

- **无审查经验沉淀机制。** 每次审查发现的新风险模式、误判记录、行业特定规则，没有结构化地写回系统。compound-engineering 的 `/ce:compound` 模式直接适用：审查完成后，将有价值的发现沉淀为规则条目（带分类、严重度、适用场景标签），下次审查同类合同时自动检索。
- **规则无版本追踪。** `default_rules.yaml` 被修改后，旧版本就丢了。法务加了一条新规则引入了误报，没法 diff 回去看是哪条引起的。
- **无 Hook 机制。** 审查流程的关键节点（审查开始、维度完成、报告生成）没有暴露 hook 点，无法挂载自定义逻辑（如通知、日志、规则自动更新）。

### 可落地的改进

最小可行方案：在 `default_rules.yaml` 的每条规则上加 `added_date` 和 `source`（人工添加 / 审查沉淀）字段。审查报告生成时，如果某个发现不属于已有规则（即 LLM 自行发现的新风险模式），在报告末尾标记为"候选新规则"，法务确认后写入 YAML。这不需要改核心引擎代码，只是规则文件格式和报告模板的变化。

## 六、错误与恢复：出错后系统凭什么还能继续

### 开源项目怎么说

harness-books 第 6 章的核心观点是反直觉的：**"错误路径是主路径。"** 在生产环境里，agent 系统遇到错误的频率远高于正常完成的频率 — 网络超时、模型拒绝、JSON 格式错误、上下文溢出，这些不是异常情况，是常态。因此错误处理不是"边缘 case"，是系统架构的核心。

具体设计模式：
- **分级恢复** — 不同错误严重度走不同恢复路径（重试 / 降级 / 熔断 / 上报）
- **熔断器** — 连续失败 N 次后停止重试，防止雪崩。Claude Code 的 autoCompact 就有"连续 3 次失败停止压缩"的熔断
- **恢复循环限制** — 重试有次数上限，防止"错误 → 重试 → 又错 → 又重试"的死循环
- **优雅降级** — 主模型不可用时自动切换到备用模型或更简单的策略

superpowers 的"3 次失败架构规则"更激进：**同一个问题修了 3 次还没解决，停下来，问题不在修复方法上，在架构上。** 不要继续打补丁，要退一步质疑前提。

### 我们做到了什么

**这是我们做得相对扎实的领域。**

1. **120 秒超时**（`client.py:18`）— LLM 调用不会无限挂起
2. **三级 fallback 链**（`retriever.py:87-103`）— `precomputed → runtime_embed → None`，每一级失败了有下一级接住
3. **防御性 JSON 解析**（`client.py:184-205`）— `_extract_tool_result()` 先试 `tool_calls`，再试 `json.loads(content)`，最后包装为 `raw_response`。三层防御，确保不会因为模型返回格式不对就崩溃
4. **维度级异常隔离**（`engine.py:101` 的 `return_exceptions=True`）— 一个维度崩了不影响其他维度。`engine.py:106-109` 的 `isinstance(result, Exception)` 检查让失败维度被记录为 `success=False` 而非整个审查崩溃

### 我们缺什么

- **无指数退避重试。** LLM 调用失败后直接报错，没有重试。对于网络抖动或模型限流（429），一次重试往往就能成功。harness-books 建议的模式是：立即重试 → 1 秒 → 2 秒 → 4 秒，最多 3-4 次。
- **无熔断器。** 如果 Ollama 服务挂了，每个审查维度、每个 chunk 都会各自超时 120 秒后才失败。4 个维度 × N 个 chunk = 很长的等待时间。应该在第一次连接失败后就标记"Ollama 不可用"，后续调用直接 fail-fast。
- **无"死信"处理。** 某个 chunk 审查失败后，这个信息丢失了（`success=False` + 错误信息）。没有机制让失败的 chunk 被单独重新审查，也没有在最终报告里明确告知用户"以下段落未能完成审查"。
- **superpowers 的 3 次失败规则缺失。** 如果同一类条款（比如保密条款）反复出错，我们没有机制识别这个模式。应该有：同一维度连续 3 次失败 → 升级为 BLOCKED，建议人工审查，而不是继续浪费 token。

### 可落地的改进

在 `LLMClient.complete()` 里加指数退避：对于 `429`（限流）和 `5xx`（服务端错误）重试最多 3 次，间隔 1s/2s/4s。对于 `4xx`（请求错误）不重试。这是最小改动、最大收益的改进。

## 七、多代理与验证：用分工管理不确定性

### 开源项目怎么说

harness-books 第 7 章的核心论点：**多 agent 的真正价值不是"多个人干活快"，而是"用分工隔离不确定性"。** 每个 agent 有自己的上下文边界，一个 agent 的错误不会污染另一个 agent 的判断。

但综合（synthesis）不能外包 — 最终把各 agent 结果整合为一致结论的工作，必须由一个知道全局的角色来做。"我让 agent 汇总了其他 agent 的结果" 听起来合理，但如果汇总 agent 不理解各结果之间的矛盾，汇总就是垃圾。

compound-engineering 的 7 persona 并行审查模式是最精妙的实践。不只是"多个人审查"，而是整个结果处理流水线：

```
7 个 persona 各自独立审查
  → 验证（每个发现是否有证据支撑）
  → 抑制（低置信度结果降级）
  → 去重（相似发现合并）
  → 交叉印证（多 persona 同时发现的问题提升优先级）
  → 矛盾消解（不同 persona 结论冲突时的处理）
  → 路由（有明确修复的自动处理，需要判断的交给人）
```

superpowers 的两阶段审查更直接地表达了一个不信任原则：**"实施者完成得可疑地快，报告可能不完整，你必须独立验证一切。"** 第一阶段是规格合规审查（做了该做的吗？），第二阶段是质量审查（做得好不好？）。审查者被明确告知不要信任实施者的自我报告。

### 我们做到了什么

1. **四维度并发审查**（`engine.py:86-101`）— risk_analysis、compliance、completeness、term_fairness 并发执行，`asyncio.gather` 的 `return_exceptions=True` 确保异常隔离
2. **SequenceMatcher 去重**（`aggregator.py:8-29`）— 阈值 0.85，加上子串包含检测（`shorter in longer → 1.0`），防止同一问题被多维度重复报告
3. **量化评分**（`aggregator.py:43-49`）— 整体风险评分 0-100，有明确的标签映射

### 我们缺什么

- **维度间无协调。** 四个维度各跑各的，互不知道对方发现了什么。如果 risk_analysis 发现了一个严重的违约金不对等条款，compliance 维度不知道这件事 — 它可能会漏掉相关的合规问题（比如违约金是否超出法定上限）。compound-engineering 的"交叉印证"思路在这里很有价值：某个风险被多个维度同时发现，应该自动提升优先级。
- **无独立验证阶段。** 四个维度跑完就出报告了，没有 superpowers 式的"第二阶段审查"。没有人检查：这些发现有引用原文吗？不同维度的结论矛盾吗？有没有遗漏？
- **无 audit trail。** 审查过程没有详细日志。每个维度看了哪些 chunk？用了哪些规则？LLM 返回了什么原始结果？如果法务质疑某个发现，我们没法回溯。

### 可落地的改进

**最值得做的是"独立验证阶段"。** 在四个维度跑完、去重之后，加一个验证步骤：

1. 检查每个 RiskFinding 的 `clause_text` 是否在原文中能找到（防止 LLM 幻觉引用）
2. 检查不同维度的结论是否矛盾（比如 term_fairness 说"条款公平"但 risk_analysis 说"该条款高风险"）
3. 对被多个维度同时标记的问题提升 severity

这不需要额外的 LLM 调用 — 第 1 步是字符串匹配，第 2、3 步是规则逻辑。成本低，但能显著提升报告可信度。

## 八、验证铁律在法务场景的特殊意义

### 为什么这一节单独列出

superpowers 项目有一套"验证铁律"体系，在软件开发场景里它保证的是代码质量。但映射到法务场景，这套体系的意义被放大了 — 因为法务审查的后果更严重，法务人员对系统的信任更脆弱。

### 核心映射

**superpowers 的验证铁律：**
1. 确定验证命令 → 2. 全新执行（不依赖缓存）→ 3. 读完整输出 → 4. 确认输出支持声明 → 5. 才可声明完成

**映射到法务审查：**
1. 确定审查标准（规则 + 维度）→ 2. 独立执行（不受之前审查的偏见影响）→ 3. 读完整审查结果 → 4. 确认每个发现有原文依据 → 5. 才可输出报告

**superpowers 的反合理化护栏：** 预设了 AI agent 的常见借口 — "太简单不需要测试""测试通过了所以代码是对的""我之后补测试" — 每条对应驳斥，从机制上阻止跳过验证。

**映射到法务审查的常见跳过理由：**

| 软件开发中的借口 | 法务审查中的对应 | 为什么危险 |
|----------------|----------------|----------|
| "太简单不需要测试" | "这个条款是标准的，不用审" | 标准条款被修改一个数字就变成不标准 |
| "测试通过了所以代码是对的" | "LLM 说没问题所以没问题" | LLM 漏检不代表没有风险 |
| "我之后补测试" | "先出报告，有问题后面修" | 报告出了就会被用于决策，没有"后面" |
| "改动太小不会有影响" | "只是措辞变化，实质不变" | 法律文本的措辞就是实质 |

### 可落地的改进

在审查报告模板里强制要求每个 RiskFinding 附带 `evidence_quote`（从合同原文直接引用的字段）。如果 LLM 的结构化输出里 `clause_text` 为空或长度不合理（比如少于 10 个字符），该 finding 自动降级为 INFO 并标注"缺乏原文依据"。

这不是完美方案（LLM 可能编造看起来合理的原文引用），但它至少建立了"无证据不可信"的机制基线。后续可以用第七节提到的字符串匹配来验证引用的真实性。

## 九、如果只改三件事

三个项目加起来的启发太多了。如果只能改三件事，优先级如下：

### P0：指数退避 + 熔断器

**来源：** harness-books 第 6 章"错误路径是主路径"

**理由：** 这是最低成本、最高确定性的改进。不涉及架构变更，不需要新增模块，只需要修改 `LLMClient.complete()` 和 `retriever._get_embedding()`。效果是立竿见影的 — 网络抖动不再导致审查失败，Ollama 挂掉不再导致所有维度超时。

**做法：**
- `LLMClient.complete()` 对 429/5xx 重试 3 次，间隔 1s/2s/4s
- 新增 `CircuitBreaker` 类：连续 3 次失败 → 打开（直接 fail-fast）→ 30 秒后半开（试一次）→ 成功则关闭
- 在 `retriever._get_embedding()` 的 runtime embedding fallback 路径也加上

### P1：独立验证阶段

**来源：** superpowers "实施者的自我报告不可信" + compound-engineering 交叉印证流水线

**理由：** 这是提升法务信任的关键。目前法务拿到报告后，需要自己去验证每个发现是否靠谱。如果系统自己先做一轮验证，法务的工作量和疑虑都会降低。

**做法：**
- 在 `engine.py` 的 `review()` 方法里，`asyncio.gather` 之后、返回报告之前，插入验证步骤
- 验证内容：(1) clause_text 在原文中存在 (2) 不同维度无矛盾 (3) 多维度交叉印证的发现提升 severity
- 不需要额外 LLM 调用，纯逻辑验证

### P2：审查经验沉淀

**来源：** compound-engineering 的 `/ce:compound` 模式

**理由：** 这是让系统越用越好的关键机制。目前每次审查是独立的，好的发现不会改进下一次审查。虽然改动范围较大（涉及规则格式、报告模板、可能的新模块），但回报也最大 — 它把系统从"工具"变成了"学习中的助手"。

**做法：**
- 规则 YAML 格式扩展：`added_date`、`source`（manual / auto_discovered）、`contract_type`（适用场景）
- 报告模板增加"候选新规则"段
- 长期：新增 `learnings/` 目录，按行业/合同类型存储审查沉淀，审查前自动检索

## 十、我们是谁：一段定位反思

harness-books Book 2 第 8 章描述了三种团队画像：

- **Claude Code 风格** — 适应性强、现场经验沉淀、灵活应对新问题，但知识散落、难以审计
- **Codex 风格** — 制度化、可复制、可审计，但学习成本高、适应新场景慢
- **混合风格** — 取两者所长，关键决策有制度保障，日常运作保持灵活

**我们目前更像 Claude Code 风格。** 系统能快速适应不同的合同类型（通过修改 YAML 规则和 Jinja2 模板），fallback 链在运行时灵活降级，分阶段加载是实际运行中发现内存问题后的"现场解法"。这些都是好的。

**但法务场景对"可审计"的要求比软件开发高。** 一个代码审查工具漏了一个 style issue，没人追责。一个合同审查工具漏了一个风险条款，可能要上法庭解释"你们系统为什么没审出来"。

所以结论不是二选一，而是明确分界线：

| 必须显式化（Codex 式） | 可以留给运行时（Claude Code 式） |
|----------------------|-------------------------------|
| 规则优先级和冲突消解 | fallback 链的降级策略 |
| 审查 audit trail | 模型选择和 API 端点 |
| 验证阶段的门控条件 | chunk 大小和并发数 |
| 经验沉淀的格式和分类 | 缓存策略 |

这份文档不是终点。它是我们第一次用一个系统性的框架审视自己的系统。三个开源项目帮我们完成了"知道自己不知道什么"这一步。

下一步是动手。从 P0 开始。
