# Claude Code 深度解析：远程智能体如何读取本地代码、隐私保护、文件格式兼容性，以及普通人如何选择交互方式

## 1. 远程智能体如何读取本地代码？——"云端大脑 + 本地双手"架构

### 核心概念：大脑在云端，手脚在本地

Claude Code 的架构可以用一个比喻来理解：

> 想象一位远程专家通过视频通话指导你修电脑。专家（大脑）在远方思考和决策，但所有实际操作——打开文件夹、查看文件、修改配置——都由你（本地的手）来执行。专家永远不会直接接触你的电脑，他只能通过你的眼睛看到他需要的东西。

在 Claude Code 中：
- **云端（Anthropic 服务器）**：运行大语言模型（Claude），负责理解需求、推理、规划、决定下一步该做什么
- **本地（你的机器）**：运行 Claude Code CLI 客户端，负责**实际执行所有工具操作**——搜索文件、读取代码、编辑文件、运行命令

**关键事实：所有工具都在你的本地机器上执行，不是在云端执行。**

### 通信架构：请求-响应循环

```
┌─────────────────┐          HTTPS/API          ┌─────────────────┐
│                 │  ──── 用户消息 + 历史 ────>  │                 │
│   你的电脑       │                              │  Anthropic 云端  │
│   (CLI 客户端)   │  <── 模型响应/工具调用 ────  │  (Claude 模型)   │
│                 │                              │                 │
│  ┌───────────┐  │                              │  思考 → 决策：   │
│  │ Grep 工具  │  │                              │  "我需要读取    │
│  │ Glob 工具  │  │                              │   src/auth.ts"  │
│  │ Read 工具  │  │                              │                 │
│  │ Edit 工具  │  │                              └─────────────────┘
│  │ Bash 工具  │  │
│  └───────────┘  │
└─────────────────┘
```

### 一次工具调用的完整生命周期

以"读取 `src/auth.ts` 文件"为例，完整流程如下：

**第 1 步：用户提问**
```
你在终端输入："帮我看看认证模块的实现逻辑"
```

**第 2 步：CLI 客户端发送请求到云端**
```
CLI 将你的消息 + 对话历史打包，通过 HTTPS 发送到 Anthropic API
（使用流式传输 streaming，这样模型的回复可以实时显示）
```

**第 3 步：云端模型思考并决定使用工具**
```
Claude 模型分析你的需求，决定：
"我需要先找到认证相关的文件，让我用 Grep 搜索一下"

模型返回一个工具调用指令（不是直接执行！）：
{
  "tool": "Grep",
  "parameters": { "pattern": "auth", "glob": "**/*.ts" }
}
```

**第 4 步：CLI 客户端在本地执行工具**
```
CLI 客户端接收到工具调用指令后：
1. 在你的本地机器上运行 ripgrep 搜索
2. 搜索结果留在本地内存中
3. CLI 将搜索结果发送回云端，作为下一轮对话的输入
```

**第 5 步：模型根据结果继续推理**
```
Claude 看到搜索结果，发现 src/auth.ts 是关键文件
再次返回工具调用指令：
{
  "tool": "Read",
  "parameters": { "file_path": "/project/src/auth.ts" }
}
```

**第 6 步：CLI 再次在本地执行**
```
CLI 读取本地文件内容，将文件内容发送回云端
```

**第 7 步：模型生成最终回答**
```
Claude 基于读到的代码内容，生成对认证逻辑的解释
这个文本回复通过流式传输实时显示在你的终端中
```

### 关键要点

1. **模型从不直接访问你的文件系统** —— 它只能通过发出工具调用"请求"来间接获取信息
2. **CLI 客户端是"守门人"** —— 它决定是否执行模型的工具调用（你可以配置权限模式来控制）
3. **每次只传输必要的内容** —— 模型不会一次性获取整个代码库，而是逐步搜索、逐个文件读取
4. **通信通过 Anthropic API** —— 使用 HTTPS 加密传输，本质上和你在 claude.ai 网页上聊天是同一个 API

## 2. 隐私保护是如何实现的？

### Claude Code 的隐私模型

Claude Code 的隐私保护基于一个核心原则：**最小化数据传输**。

#### 什么数据会发送到云端？

| 数据类型 | 是否发送 | 说明 |
|----------|---------|------|
| 你的提问/指令 | ✅ 是 | 模型需要理解你的需求 |
| 对话历史 | ✅ 是 | 保持上下文连贯性 |
| 工具执行结果 | ✅ 是 | 搜索结果、文件内容——但只是**模型主动请求的那些** |
| CLAUDE.md 内容 | ✅ 是 | 作为系统提示的一部分 |
| 整个代码库 | ❌ 否 | 从不整体上传 |
| 未被请求的文件 | ❌ 否 | 只有模型明确"要求读取"的文件才会发送 |
| 你的 SSH 密钥/凭证 | ❌ 否 | CLI 不会读取或发送认证凭据 |
| .claudeignore 中的文件 | ❌ 否 | 被明确排除的文件连工具都无法访问 |

#### "按需搜索"为何比"全量索引"更保护隐私？

对比两种架构的隐私差异：

**Claude Code（按需搜索）**：
```
代码库 1000 个文件
  → 模型搜索后决定读取 3 个文件
  → 只有这 3 个文件的内容被发送到云端
  → 其余 997 个文件从未离开你的机器
```

**Cursor/Windsurf（全量索引）**：
```
代码库 1000 个文件
  → 全部 1000 个文件被切块、嵌入
  → 嵌入向量被发送到云端（Cursor: Turbopuffer, Windsurf: 其服务器）
  → 虽然不是代码原文，但向量中可能"残留"代码语义信息
```

#### 具体的隐私保障措施

1. **本地工具执行**：所有搜索和读取操作在你的机器上执行，结果先由 CLI 客户端处理
2. **.claudeignore**：类似 `.gitignore`，可以明确排除敏感文件（如 `.env`、`credentials.json`），这些文件对 Claude 完全不可见
3. **权限模式**：
   - `--allowedTools`：白名单模式，只允许使用指定的工具
   - 交互式确认：默认情况下，文件写入和命令执行会请求你的批准
4. **提示缓存隔离**：Anthropic 的提示缓存是按组织隔离的，你的缓存数据不会泄露给其他用户
5. **无持久化存储**：Anthropic 声明不会使用通过 API 发送的数据来训练模型（需开启数据保留豁免）
6. **传输加密**：所有通信通过 HTTPS/TLS 加密

#### 与 Cursor/Windsurf 的隐私对比

| 隐私维度 | Claude Code | Cursor | Windsurf |
|----------|-------------|--------|----------|
| **代码是否离开本地** | 仅被请求的片段 | 嵌入向量上传到 Turbopuffer | 代码片段发送到服务器嵌入 |
| **云端持久化** | 不持久化代码 | 向量持久存储在云端 | 向量返回本地存储 |
| **全量上传** | 从不 | 初始索引时全量处理 | 初始索引时逐块处理 |
| **文件排除** | .claudeignore | .cursorignore | 设置中配置 |
| **零信任选项** | 可完全离线（本地模型） | 需要云端索引 | 需要云端嵌入计算 |

## 3. 代码格式对智能体是否天然友好？——从代码到富文本的鸿沟

### 为什么源代码是 AI 智能体的"理想食物"？

源代码具有一系列对 AI 智能体极度友好的特性：

| 特性 | 为什么对智能体友好 | 举例 |
|------|-------------------|------|
| **纯文本** | 可以直接被任何工具读取，无需解码或转换 | `.py`、`.js`、`.ts` 文件用任何文本编辑器都能打开 |
| **行结构** | 每一行通常是一个独立的语句，便于定位和引用 | `grep` 可以精确找到第 42 行的 `def authenticate()` |
| **结构化语法** | 有明确的语法规则，可以被解析器（parser）精确分析 | tree-sitter 可以把代码解析成语法树（AST） |
| **可搜索性** | 函数名、变量名、类名是唯一标识符，搜索几乎不会有歧义 | 搜索 `handleLogin` 找到的一定是那个函数 |
| **自文档化** | 好的代码本身就是文档——命名规范、注释、类型声明 | `function calculateTotalPrice(items: CartItem[]): number` |
| **差异友好** | 可以精确比较两个版本之间的差异（diff） | `git diff` 能精确到某一行的某个字符的变化 |
| **模式化** | 同一种语言的代码遵循相同的模式，AI 在训练中见过海量代码 | Python 的 `class/def/import` 模式高度一致 |

总结：**代码是纯文本 + 结构化 + 可搜索 + 模式化的，这让 Grep/Glob/Read 三板斧就能高效工作。**

### 富文本格式：一道"翻译墙"

当工作区中是 `.docx`、`.pptx`、`.xlsx`、`.pdf` 或图片时，情况完全不同：

#### 各种富文本格式的本质

```
源代码文件 (.py, .js, .ts)
  └── 本质：纯文本 → 直接可读 ✅

Word 文档 (.docx)
  └── 本质：ZIP 压缩包 → 内含 XML 文件 + 媒体文件
  └── 用 grep 搜索？❌ 会得到乱码

PowerPoint (.pptx)
  └── 本质：ZIP 压缩包 → 内含 XML + 图片 + 布局信息
  └── 用 Read 工具读取？❌ 会得到二进制数据

Excel (.xlsx)
  └── 本质：ZIP 压缩包 → 内含 XML（数据分散在多个文件中）
  └── 公式、格式、图表信息编码在 XML 中

PDF (.pdf)
  └── 本质：二进制格式，包含文本流、字体、图形指令
  └── 文本可能被打散成单个字符+坐标的组合

图片 (.png, .jpg)
  └── 本质：像素矩阵的二进制编码
  └── 完全无法用文本搜索工具处理
```

#### Claude Code 对各种格式的支持能力

| 格式 | CLI 支持情况 | 具体能力 |
|------|-------------|---------|
| **源代码**（.py, .js, .ts 等） | ✅ 完全支持 | Read/Grep/Glob/Edit 全部可用 |
| **PDF** | ✅ 支持读取 | Read 工具可以读取 PDF，支持指定页码范围（如 `pages: "1-5"`）；大型 PDF（>10页）必须指定页码 |
| **图片**（.png, .jpg 等） | ✅ 支持查看 | Read 工具可以读取图片，Claude 作为多模态模型可以理解图片内容（截图、UI 设计稿等） |
| **Jupyter Notebook**（.ipynb） | ✅ 支持读取和编辑 | Read 工具返回所有单元格及其输出；NotebookEdit 工具可以编辑单元格 |
| **Word**（.docx） | ⚠️ 有限支持 | 需要借助 Bash 工具调用 `python-docx` 或 `pandoc` 等外部工具来提取文本 |
| **PowerPoint**（.pptx） | ⚠️ 有限支持 | 需要外部工具（如 `python-pptx`）提取内容 |
| **Excel**（.xlsx） | ⚠️ 有限支持 | 需要外部工具（如 `openpyxl`、`pandas`）读取数据 |

#### 关键区别：原生支持 vs 需要工具链

```
原生支持（Read 工具直接处理）：
  源代码 → 直接读取文本 → 完美 ✅
  PDF    → 内置解析器提取文本 → 良好 ✅
  图片   → 多模态视觉理解 → 良好 ✅
  .ipynb → 内置 JSON 解析 → 良好 ✅

需要工具链（通过 Bash 调用外部工具）：
  .docx → bash: python -c "import docx; ..." → 可行但繁琐 ⚠️
  .xlsx → bash: python -c "import openpyxl; ..." → 可行但繁琐 ⚠️
  .pptx → bash: python -c "import pptx; ..." → 可行但繁琐 ⚠️
```

### 为什么这种差异很重要？

Claude Code 的"智能体搜索"（Grep + Glob + Read）是为**纯文本世界**设计的。当你的工作区全是代码文件时，这套工具链如鱼得水。但当工作区是富文本文档时：

1. **Grep 无法搜索** —— `.docx` 是二进制压缩包，关键词搜索失效
2. **Glob 能找到文件** —— 但找到后无法直接读取内容
3. **Read 可能返回乱码** —— 二进制文件不能作为纯文本读取
4. **Edit 无法使用** —— 不能用文本替换来编辑 Word 文档

**这正是 Cowork 模式存在的核心原因之一**——它为非代码文件的处理提供了原生支持。

## 4. Claude 应用中的三种模式：Chat、Code、Cowork

### 为什么会有三种模式？

Anthropic 在 Claude 桌面应用中提供了三种交互模式，每种针对不同的用户群体和使用场景：

```
┌──────────────────────────────────────────────────────────┐
│                    Claude 桌面应用                        │
│                                                          │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐            │
│  │   Chat   │   │  Cowork  │   │   Code   │            │
│  │  聊天模式 │   │ 协作模式  │   │  编码模式 │            │
│  │          │   │          │   │          │            │
│  │ 对话问答  │   │ 知识工作  │   │ 软件开发  │            │
│  │ 头脑风暴  │   │ 文档处理  │   │ 代码编写  │            │
│  │ 写作辅助  │   │ 数据分析  │   │ 调试修复  │            │
│  │ 学习研究  │   │ 项目管理  │   │ 部署发布  │            │
│  │          │   │          │   │          │            │
│  │ 👤 所有人 │   │ 👤 知识   │   │ 👤 开发者 │            │
│  │          │   │   工作者  │   │          │            │
│  └──────────┘   └──────────┘   └──────────┘            │
│                                                          │
│  无需设置 ←──── 复杂度/能力 ────→ 完全控制               │
└──────────────────────────────────────────────────────────┘
```

### Chat 模式（聊天）

- **目标用户**：所有人
- **核心能力**：对话问答、写作、翻译、头脑风暴、学习
- **文件访问**：不直接访问本地文件（可以上传文件到对话中）
- **技术门槛**：零
- **适用场景**：快速提问、一次性任务、不需要持续文件操作的工作

### Cowork 模式（协作）

- **目标用户**：知识工作者——产品经理、分析师、运营、研究人员、非技术团队
- **发布时间**：2026 年 1 月 16 日（macOS），2 月 10 日（Windows）
- **核心能力**：
  - 直接读写本地文件（你共享的文件夹范围内）
  - **生成专业文档**：Excel（含工作公式）、PowerPoint、格式化文档
  - **子智能体协调**：将复杂工作拆分为多个并行工作流
  - **定时任务**：可以安排后续自动执行
  - **插件连接**：Slack、Notion、Figma 等
- **执行环境**：隔离的虚拟机（更安全，权限受限）
- **技术门槛**：极低——不需要终端知识，全程图形界面

**Cowork 诞生的原因**：Anthropic 发布 Claude Code 后，发现开发者们不仅用它写代码，还大量用于研究、文档创建、项目管理等非编码任务。这说明存在一个巨大的需求——把智能体式的工作能力带给非技术用户。Cowork 就是为此而生的。

### Code 模式（编码）

Code 模式在桌面应用/网页中提供了与 CLI 相同的核心引擎，但增加了图形界面：

- **目标用户**：开发者
- **两种环境**：
  - **本地模式**：选择本地文件夹，Claude 直接操作你的文件
  - **远程模式**：连接 GitHub 仓库，Claude 在 Anthropic 托管的云端沙盒中工作
- **核心能力**：编写代码、修复 bug、重构、运行测试、创建 PR、部署
- **执行环境**：直接访问项目（本地模式）或云端沙盒（远程模式）
- **技术门槛**：中等——需要理解代码和项目结构

### 桌面/网页 Code 模式 vs CLI 的区别

| 维度 | CLI（终端） | 桌面/网页 Code 模式 |
|------|-----------|-------------------|
| **文件系统访问** | 完全访问本地文件系统 | 本地模式：选定文件夹；远程模式：沙盒 |
| **SSH/凭证** | 可以使用本地 SSH 密钥 | 沙盒环境无法访问本地凭证 |
| **并行任务** | 单一会话 | 支持跨多个仓库并行执行 |
| **Unix 组合** | 可以与管道、CI 等组合 | 图形界面，不可组合 |
| **文件大小限制** | 无硬性限制 | 网页版单文件 1MB、仓库 500MB |
| **会话持久性** | 关闭终端即结束 | 远程会话可持久化 |
| **会话互通** | 可用 `--remote` 推送到云端 | 可用 `/teleport` 拉回本地 CLI |
| **适合谁** | 熟练使用终端的开发者 | 偏好图形界面的开发者 |

**底层引擎完全相同**——使用相同的模型、相同的工具、读取相同的 CLAUDE.md 配置文件。

### VS Code / JetBrains 插件

插件模式在 IDE 内部集成 Claude Code：
- 底层封装的是同一个 CLI 引擎
- 额外感知：当前打开的文件、光标位置、选中的代码
- 支持 `@` 引用文件和行范围
- 对于已经生活在 IDE 中的开发者来说，这是最自然的交互方式

## 5. 三种模式的设计哲学——为什么要做这种区分？

### 不同人群有不同的"数字原生格式"

这是理解三种模式最关键的洞察：

```
开发者的世界：                    普通人的世界：
├── src/                         ├── 项目方案.docx
│   ├── auth.ts                  ├── 季度报告.xlsx
│   ├── database.py              ├── 产品演示.pptx
│   └── api/routes.js            ├── 会议纪要.pdf
├── tests/                       ├── 设计稿.png
├── package.json                 └── 客户反馈.csv
└── README.md

纯文本 + 结构化                   富文本 + 视觉化
代码就是"文档"                    文档才是"工作成果"
终端是自然栖息地                   Word/Excel 是自然栖息地
```

**Claude Code（CLI/插件/桌面 Code 模式）** 是为左边的世界优化的——它的搜索工具、编辑工具、差异追踪都完美适配纯文本。

**Cowork** 是为右边的世界优化的——它能生成 Excel（含工作公式）、PowerPoint、格式化文档，并且在隔离环境中安全地操作这些文件。

**Chat** 是两个世界的公共入口——不需要文件操作时，纯对话就够了。

### 这是一条"渐进式能力通道"

```
Chat ──→ Cowork ──→ Code
 │          │          │
 │          │          └── 最强能力：完全控制文件系统和终端
 │          │              最高门槛：需要理解代码
 │          │
 │          └── 中等能力：文件读写 + 文档生成 + 子智能体
 │              低门槛：图形界面，无需技术知识
 │
 └── 基础能力：对话问答
     零门槛：会打字就行
```

Anthropic 的策略是：**让每个人都能从自己的舒适区出发，逐步体验到越来越强大的 AI 能力。**

## 6. 普通人应该如何选择？——实用决策指南

### 快速决策树

```
你的任务是什么？
│
├── 只是问个问题 / 写篇文章 / 翻译 / 头脑风暴
│   └── → 用 Chat 模式 ✅
│
├── 需要处理文件：整理文档、分析数据、制作报告、研究调查
│   ├── 你熟悉终端/命令行吗？
│   │   ├── 不熟悉 → 用 Cowork 模式 ✅
│   │   └── 熟悉 → Cowork 或 CLI 都可以，看任务复杂度
│   └── 需要生成 Word/Excel/PPT？
│       └── → 用 Cowork 模式 ✅（Code 模式需要额外脚本才能生成这些格式）
│
├── 需要写代码 / 修 bug / 部署项目
│   ├── 你习惯用终端吗？
│   │   ├── 是 → 用 CLI (Claude Code) ✅
│   │   └── 否 → 用桌面/网页 Code 模式 或 VS Code 插件 ✅
│   └── 需要操作远程仓库但不想在本地 clone？
│       └── → 用桌面/网页 Code 模式的远程环境 ✅
│
└── 大型复杂项目：需要同时研究 + 写代码 + 生成文档
    └── → 混合使用：Cowork 做研究和文档，Code 做代码 ✅
```

### 按用户角色推荐

| 你的角色 | 推荐模式 | 原因 |
|----------|---------|------|
| **学生/普通用户** | Chat | 学习、写作、翻译、问答——不需要文件操作 |
| **产品经理** | Cowork | 需求文档、竞品分析、用户故事——生成专业文档 |
| **数据分析师** | Cowork | 数据处理、报告生成——可以生成带公式的 Excel |
| **研究人员** | Cowork + Chat | 文献整理、论文辅助、数据可视化 |
| **运营/市场** | Cowork | 营销方案、数据报告、演示文稿 |
| **前端开发者** | VS Code 插件 | 在 IDE 中无缝编码，看到实时预览 |
| **后端开发者** | CLI | 终端中最高效，可与 git/docker/CI 无缝组合 |
| **全栈开发者** | CLI + 桌面 Code | CLI 做本地开发，桌面 Code 做远程并行任务 |
| **技术领导** | 桌面 Code 模式 | 审查 PR、理解代码架构——图形界面更直观 |

### 核心建议

1. **不需要为了"更强大"而勉强自己用 CLI** —— 三种模式底层使用相同的 Claude 模型，智能水平完全一致。差异仅在于交互方式和工具访问范围。

2. **从 Cowork 开始是非技术用户的最佳起点** —— 它提供了智能体式工作流的核心价值（自主规划、分步执行、文件操作），同时没有任何技术门槛。

3. **Code 模式不是 Cowork 的"升级版"** —— 它们是为不同工作内容设计的。写代码用 Code，做知识工作用 Cowork。不存在"更高级"的问题。

4. **混合使用是完全正常的** —— 你可以在 Cowork 中做调研、整理需求，然后在 Code 中实现代码，最后回到 Cowork 生成项目文档。

## 7. 总结：一张全景图

```
                        Claude 生态系统
                             │
          ┌──────────────────┼──────────────────┐
          │                  │                  │
      Chat 模式          Cowork 模式         Code 模式
      (对话)             (协作)              (编码)
          │                  │                  │
      纯对话             文件操作             代码操作
      无工具             隔离环境             完全访问
          │                  │                  │
          │                  │         ┌────────┼────────┐
          │                  │         │        │        │
          │                  │      桌面/网页  VS Code   CLI
          │                  │      Code模式   插件      终端
          │                  │         │        │        │
          │                  │      图形界面  IDE内嵌  命令行
          │                  │      可远程    可感知    可组合
          │                  │               编辑器
          │                  │
     ─────┴──────────────────┴─────────────────────────────
     所有人              知识工作者                 开发者
     零门槛              低门槛                     需技术背景

     共同点：相同的 Claude 模型 | 相同的智能水平 | 相同的推理能力
     差异点：交互方式 | 工具权限 | 执行环境 | 目标文件格式
```
