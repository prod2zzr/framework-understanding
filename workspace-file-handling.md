# AI 编码工具如何"认识"你的代码？——工作区文件处理方式对比

> **一句话总结**：Cursor 和 Windsurf 会在你打开项目时自动扫描所有文件、生成"语义索引"（向量化），之后能"凭理解"找代码；Claude Code 和 Gemini CLI **不会**自动扫描，而是像人一样"现找现看"——需要时再搜索、再阅读。两种方式各有利弊。

---

## 零、先理解几个关键概念

### 什么是"扫描和预处理（向量化）"？

想象你搬进一间新图书馆。有两种找书方式：

- **方式 A（建索引）**：先花几个小时给每本书编一张"内容摘要卡"，按主题分类存进卡片柜。以后找书时，先查卡片柜，直接定位到书架。——这就是 **Cursor / Windsurf** 的做法。
- **方式 B（现场翻找）**：不建卡片柜，需要时直接去书架上按书名搜、翻目录、查关键词。——这就是 **Claude Code / Gemini CLI** 的做法。

技术上，"方式 A"的核心步骤叫做**向量化（Embedding）**：把每段代码转换成一串数字（向量），这串数字能捕捉代码的"含义"。之后搜索时，把你的问题也变成一串数字，然后找"含义最接近"的代码片段。这整个流程叫做 **RAG（检索增强生成）**。

### 什么是"上下文窗口"？

AI 每次回答问题时，能"看到"的信息总量是有限的——就像人的短期记忆有限一样。这个限制叫做**上下文窗口**。如果你的项目有上万个文件，AI 不可能一次全部读完，所以必须有办法筛选出"最相关"的文件片段塞进这个窗口。

## 一、四种工具的处理方式一览

| | Cursor | Windsurf | Claude Code | Gemini CLI |
|---|---|---|---|---|
| **打开项目时会自动扫描全部文件吗？** | 会 | 会 | **不会** | **不会** |
| **会生成向量索引吗？** | 会（上传到云端计算） | 会（本地计算为主） | **不会** | **不会** |
| **找代码的方式** | 语义搜索（按"含义"找） | 语义搜索 + 意图追踪 | 关键词搜索 + 逐文件阅读 | 关键词搜索 + 逐文件阅读 |
| **需要用户手动提供上下文吗？** | 较少 | 最少 | **较多** | **较多** |
| **首次打开大项目的等待时间** | 较长（需要建索引） | 中等 | **几乎为零** | **几乎为零** |

## 二、会自动扫描的工具：Cursor 和 Windsurf

### Cursor 的做法

Cursor 打开项目后会自动执行以下步骤：

```
打开项目
   ↓
扫描全部文件（跳过 .gitignore 中忽略的文件）
   ↓
把每个文件切成小块（按函数、类等语法边界切割）
   ↓
把代码块加密后上传到 Cursor 的服务器
   ↓
服务器用 AI 模型计算每个代码块的"含义向量"
   ↓
向量存入专用的向量数据库（Turbopuffer）
   ↓
立即删除服务器上的代码原文，只保留向量
```

**之后你问问题时**：Cursor 把你的问题也变成向量 → 在数据库中找"含义最接近"的代码块 → 从你本地电脑读取那些代码的原文 → 连同你的问题一起发给 AI。

**增量更新**：Cursor 用一种叫"Merkle 树"的数据结构（类似文件指纹树），每 10 分钟检查哪些文件改了，只重新处理改动的部分，不用每次全量扫描。

**隐私处理**：代码原文不会长期存储在服务器上；文件名会被加密混淆；服务器只保留不可直接还原为代码的数字向量。

**适用规模**：50 万行以内的项目表现良好，超大型项目（百万行级）可能会变慢。

### Windsurf 的做法

Windsurf 的方式与 Cursor 类似，但有几个关键区别：

- **本地优先**：默认在你自己的电脑上完成索引计算，代码不离开你的机器
- **更智能的切割方式**：使用 AST（抽象语法树）技术，在函数、方法、类的边界切割代码，比简单的文本切割更精准
- **意图追踪（Cascade）**：不只是搜索相关代码，还会追踪你当前的"意图"。例如你在修一个导航 bug，它会自动把相关的页面文件和路由逻辑都找出来，不需要你手动 @ 指定
- **打开项目即开始索引**，文件变化时自动更新
- **企业版支持远程索引**，可以处理百万行级别的超大项目

## 三、不会自动扫描的工具：Claude Code 和 Gemini CLI

### Claude Code 的做法

Claude Code **不建任何索引**，也不做向量化。它的工作方式更像一个真人开发者：

```
你提出需求
   ↓
Claude 思考需要找哪些文件
   ↓
用 grep（关键词搜索）和 glob（文件名匹配）在项目中搜索
   ↓
找到可能相关的文件后，逐个阅读内容
   ↓
根据读到的内容，决定是否需要继续搜索更多文件
   ↓
收集到足够信息后，给出回答或修改代码
```

**为什么不建索引？**

这是 Anthropic 的设计哲学选择：
- Claude Code 定位为轻量级 CLI 工具，追求"打开即用"，不需要额外的服务器或数据库
- 向量数据库和嵌入模型不是 Anthropic 的核心技术方向
- 有研究（2026 年 Amazon Science 论文）表明：基于关键词的智能搜索能达到向量搜索 90% 以上的效果
- Claude 拥有 100 万 token 的超大上下文窗口，在很多场景下可以直接阅读大量文件而不需要索引辅助

**代价是什么？**

- 每次搜索和阅读文件都消耗 token（token 是按量计费的）
- 对于超大项目，可能需要多轮搜索才能找到关键代码
- AI 可能会遗漏一些语义相关但关键词不匹配的代码

### Gemini CLI 的做法

Gemini CLI 同样不做向量化索引，但它的初始步骤比 Claude Code 多一点：

```
启动 Gemini CLI
   ↓
自动生成当前目录的"文件树快照"（只看文件名和目录结构，不读内容）
   ↓
把这个文件树作为初始上下文发送给 AI（用户看不到这一步）
   ↓
你提出需求
   ↓
Gemini 根据文件树猜测哪些文件可能相关
   ↓
用搜索工具（关键词搜索、文件名匹配）找到具体文件
   ↓
逐个阅读文件内容
   ↓
给出回答
```

**和 Claude Code 的区别**：Gemini CLI 多了一个"文件树快照"步骤，相当于先看了一眼图书馆的目录，知道有哪些书，但还没翻开任何一本。这让它在猜测"该去看哪个文件"时更准一些。

**社区呼声**：Gemini CLI 的 GitHub 仓库上有多个用户请求添加向量化索引功能（Issue #2065、#5150），但目前尚未实现。社区中有人尝试在启动时提供函数/类的大纲列表，发现能显著减少搜索次数。

## 四、直接对比：两种路线的优劣

### "自动扫描 + 向量化"路线（Cursor / Windsurf）

| 优势 | 劣势 |
|------|------|
| 能按"含义"搜索，不依赖精确关键词 | 首次打开项目需要等待索引完成 |
| 找代码更快更准，尤其是大项目 | 索引过程消耗 CPU / 内存 / 网络资源 |
| 用户不需要手动指定上下文 | Cursor 需上传代码块到云端（隐私顾虑） |
| 支持"我想找类似功能的代码"这类模糊搜索 | 索引可能过时，存在几分钟的延迟 |

### "不扫描，按需搜索"路线（Claude Code / Gemini CLI）

| 优势 | 劣势 |
|------|------|
| 打开即用，零等待 | 每次搜索消耗 token（= 花钱） |
| 代码不离开你的电脑（更安全） | 可能漏掉"意思相关但关键词不同"的代码 |
| 不占用额外的 CPU / 内存 / 磁盘 | 大项目中找代码可能需要多轮搜索 |
| 不依赖外部服务器 | 需要用户更主动地提供上下文 |

## 五、用户如何应对"不自动扫描"的工具？

如果你使用 Claude Code 或 Gemini CLI，以下技巧可以弥补它们"不建索引"的不足：

### 技巧 1：写好项目说明文件

在项目根目录创建一个 `CLAUDE.md`（Claude Code 专用）或 `GEMINI.md`（Gemini CLI 专用）文件，写上：

```markdown
# 项目结构说明

这是一个 React + Node.js 的电商网站。

## 关键目录
- src/components/ — 前端 UI 组件
- src/api/ — 后端接口
- src/models/ — 数据库模型
- src/utils/auth.ts — 认证逻辑（登录、注册、Token 验证）

## 代码规范
- 使用 TypeScript
- 组件使用函数式写法
- API 路由文件以 .route.ts 结尾
```

AI 每次启动时会自动读取这个文件，相当于先给它一张"地图"。

### 技巧 2：用 @ 符号手动指定文件

在对话中直接告诉 AI 去看哪个文件：

```
请看一下 @src/utils/auth.ts 里的登录逻辑，帮我找出为什么 token 过期后没有自动刷新
```

### 技巧 3：先让 AI 熟悉项目

在开始正式工作之前，先让 AI 浏览一下项目结构：

```
请先看看这个项目的目录结构和关键文件，告诉我你对这个项目的理解
```

### 技巧 4：安装社区开发的向量搜索插件（进阶）

Claude Code 支持 MCP（模型上下文协议）扩展，社区已经开发了向量搜索插件，可以给 Claude Code 补上"自动索引"的能力：

- **Claude Context**（Zilliz 开发）：使用 BM25 + 向量混合搜索，通过 AST 智能切割代码，测试显示在同等检索质量下可减少约 40% 的 token 消耗
- **自建 ChromaDB MCP 服务器**：用开源向量数据库 ChromaDB 为项目建立本地索引
- **CocoIndex**：为多种 AI 编码工具（Claude Code、Gemini CLI、Cursor 等）提供语义代码搜索

安装方法（以 Claude Context 为例）：
```bash
# 在终端中添加 MCP 服务器
claude mcp add claude-context -- npx -y @anthropic-ai/claude-context
```

### 技巧 5：合理控制项目范围

- 用 `.gitignore` 排除无关文件（`node_modules`、构建产物等）
- 如果项目很大，考虑在子目录中启动 AI，缩小搜索范围
- 拆分对话：一次对话专注一个模块，而不是让 AI 同时处理整个项目

## 六、总结——如何选择？

| 你的情况 | 推荐工具 |
|---------|---------|
| 项目很大（几十万行），需要频繁跨文件搜索 | Cursor 或 Windsurf（自动索引更省心） |
| 在意代码隐私，不希望代码离开本机 | Claude Code CLI 或 Windsurf（本地索引） |
| 项目中小规模，偏好轻量工具 | Claude Code 或 Gemini CLI（零开销） |
| 需要在远程服务器 / CI 流水线中使用 | Claude Code CLI 或 Gemini CLI（无需 GUI） |
| 想要两全其美 | Claude Code + 社区向量搜索插件（MCP） |

**最终结论**：Claude Code 和 Gemini CLI **不会**像 Cursor / Windsurf 那样自动向量化你的代码库。这是有意为之的设计选择，不是功能缺失。对于中小型项目，"按需搜索"完全够用；对于大型项目，可以通过写好说明文件、手动指定上下文、或安装社区向量搜索插件来弥补。
