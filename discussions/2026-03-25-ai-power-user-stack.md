# Mixing Models, Switching CLIs: Anatomy of a Power User's AI Stack

> 来源：Ben Thompson ([Ben's Bites](https://www.bensbites.com/p/how-and-what-im-building-this-week)) 分享的个人技术栈，经中文推文传播后引发讨论。

---

## 1. 模型层：混用才是正道

Ben 的策略很明确：**不同任务用不同模型**。

| 任务类型 | 模型选择 | 原因 |
|---------|---------|------|
| 新功能开发、写"正经代码" | GPT 5.4 XHigh | 精准、执行力强 |
| 规划、研究、设计、非技术任务 | Opus 4.6 | 推理深度、全局视野，**always** |

### 为什么这样做有道理？

这和人类团队的分工逻辑一致：**架构师负责设计，工程师负责实现**。把同样的分工逻辑应用到模型上，是一种成熟的工作流。

- **Planning 和 Coding 是质的不同的任务。** 规划需要广度、全局思考、对上下文的整体把握；代码执行需要精确、符合 spec、模式匹配能力强。
- Ben 在 design 上用 Opus 标注了 "always"——说明这个差距并不微妙，而是明确的。
- 用最贵的模型做所有事是浪费，用最便宜的做所有事会在关键任务上牺牲质量。**混用是性价比最优解。**

## 2. CLI 层：Droid + Pi，互补而非冗余

两个 CLI 工具，各有分工：

### Droid — 重型施工
- "for when I want to build something properly"
- Missions 功能可以**自主运行数小时**，端到端实现功能
- 适合复杂、多步骤的长线任务

### Pi — 轻快迭代
- 快速、轻量
- **你自己的 instructions 对它的引导作用比其他工具更大**
- 适合快速探索、小任务、日常交互

### 关键共同点：模型随时切换

> Both let you switch from GPT ←→ Claude models (or Gemini, etc.) in one conversation.

这是整个工作流的**关键enabler**。没有同一对话内的模型切换能力，"Opus 做规划 → GPT 做执行"的策略就需要在工具间复制粘贴上下文，摩擦成本太高。

**选 CLI 工具时最该看的能力：能否在一个对话里切换模型。**

## 3. 环境层：终端为王，编辑器靠边

| 工具 | 角色 |
|-----|------|
| Ghostty | 终端模拟器（现在包含在 Cmux 内） |
| Cmux | 终端复用器 + 工作空间管理器：侧边栏管理对话、可拖拽面板、内置浏览器 |
| Zed | **仅用于查看文件**——不是主力编码界面 |

### 这说明了什么？

**编辑器被"降级"了。** Zed 是一个 viewer，不是工作流的中心。真正的工作发生在终端里——AI CLI 在终端中运行，终端复用器管理整个工作空间。

这是一个**"对话优先"的开发环境**：
- 主要界面 = 终端 + AI CLI
- 编辑器 = 辅助参考工具
- 工作空间管理 = Cmux（不是 IDE 的窗口系统）

传统的 IDE 中心开发方式正在被 **Terminal + AI CLI** 的组合取代。

---

## 4. 更大的图景

Ben 的栈不只是个人偏好，它反映了几个正在发生的趋势：

1. **模型专业化成为标配**——不再是 power user 的技巧，而是默认实践
2. **支持模型切换的 CLI 正在赢**——锁定单一模型的工具生态处于劣势
3. **"技术栈"的定义在变**——从 language / framework / database 转向 **model / CLI / terminal**
4. **IDE 正在被终端取代**——当 AI 成为主要编码伙伴时，对话界面比文件编辑界面更重要
